# Денормализованные FK: цепочка fallback'ов от достоверного к наименее достоверному

**Дата:** 2026-05-10.
**Тикет:** FEAT-016 (per-student MD-отчёт + bulk ZIP).
**Связанные коммиты:** `0b50fbf` (через week_id), `4eea3d8` (через
lesson_id), `6c2a2c1` (через content_item.external_key).

## Симптом

В FEAT-016 per-student MD-отчёт группировал домашки по
`module_number`. После phase 25 backfill'а у нас в
`pvl_homework_items` появилась колонка `module_number int`,
backfill через regex по title (`'модул[ьюяе]\s*(\d+)'`) проставил
её для **6 из ~24 домашек**. Остальные остались с `module_number IS NULL`.

В UI это выглядело как:
- группа «Модуль 1» содержит 2-3 строки (повезло, regex попал);
- группа «Без модуля» содержит почти всё остальное (12-15 строк);
- per-student MD-отчёт за «Модуль 2» получался **пустым**.

После трёх итераций фолбэков (через `week_id`, через `lesson_id`,
через `content_items.external_key`) — отчёт стал заполняться
корректно.

## Корневая причина

`module_number` в системе **денормализован** — лежит в нескольких
таблицах с разной достоверностью:

```
pvl_content_items.module_number   — самый достоверный, заполнен
                                     для всех контент-айтемов курса
pvl_course_weeks.module_number    — заполнен при создании структуры курса
pvl_course_lessons.module_number  — заполнен косвенно через week_id
pvl_homework_items.module_number  — последний по цепочке, backfill
                                     покрыл малую часть (regex по title)
```

Если у домашки `pvl_homework_items.module_number IS NULL`, то по
имеющимся FK можно дойти до источника правды разными путями:

```
homework_item
   ↓ external_key (уникальный slug)
content_item (имеет module_number) ← Best ✅
   ↓ или fallback:
homework_item.week_id → course_week (имеет module_number)
   ↓ или fallback:
homework_item.lesson_id → course_lesson → course_week (имеет module_number)
```

phase 25 backfill сделал regex'ом только по `title` —  это покрыло
домашки, которые в названии содержат «Модуль N», но **большинство
домашек называются по теме** («Рефлексия по модулю», «Анкета
обратной связи», «Письменная практика "..."») и regex их пропустил.

Слой ответственности — UI client-side, потому что:
- сам homework_item «не знает» свой модуль (NULL);
- backfill не покрыл (regex слабоват);
- альтернативные источники доступны через FK, но требуют доп.
  joins на клиенте.

## Почему так получилось

1. **Денормализация без single source of truth.** При проектировании
   курса phase 25 решили хранить `module_number` в **четырёх**
   таблицах для удобства запросов. Это типичный trade-off:
   читать быстро (один SELECT), писать сложно (поддерживать
   консистентность). Backfill попытался синхронизировать
   `homework_items` с другими, но по неполным признакам.
2. **Regex backfill — best-effort, не контракт.** Regex
   `'модул[ьюяе]\s*(\d+)'` зацепил 6 домашек из ~24 (25%). Всё
   остальное — fallback на UI. Если бы regex покрыл 100%, fallback
   не понадобился бы. Но 100% невозможен — названия намеренно
   тематические, не структурные.
3. **Контент создавался руками, без structured input.** Domain
   эксперт (Ольга) добавляет уроки и домашки через mock-данные с
   человеко-читаемыми title'ами, не думая о regex-friendly формате.
   Это **правильно** — мы не должны диктовать стиль ради backfill'а.
4. **На UI пришёл NULL — нет fallback в первой версии.** Первый
   подход в UI (commit `73b2132`) предполагал, что
   `homework_item.module_number` всегда заполнен → NULL ломал
   группировку. Это симптом «data contract not enforced» — UI
   полагался на инвариант, который не гарантирован.

## Как починили

В UI добавили **цепочку фолбэков** в порядке достоверности:

```js
// views/AdminPvlProgress.jsx, useMemo для группировки homework_items
const moduleNumber = homeworkItem.module_number
    ?? contentItemsByExternalKey.get(homeworkItem.external_key)?.module_number
    ?? courseWeeksById.get(homeworkItem.week_id)?.module_number
    ?? courseLessonsById.get(homeworkItem.lesson_id)?.course_week?.module_number
    ?? null;
```

Каждый шаг — отдельный коммит, fix-by-fix:

- `0b50fbf` — добавил fallback через `week_id` (+ загрузка
  `pvl_course_weeks` в memo).
- `4eea3d8` — добавил третий fallback через `lesson_id` (для
  домашек, которые привязаны к lesson, не к week напрямую).
- `6c2a2c1` — main fallback через `content_items.external_key`
  (по факту самый частый case).

Каждый фолбэк перенумерован: первый — наиболее достоверный
(`content_item` по external_key — у него поле точно заполнено,
один-в-один). Последний — наименее достоверный (`lesson → week →
module`, два FK прыжка).

Если все 4 источника NULL — последняя надежда `?? null` → группа
«Без модуля». Это ОК для редких edge-кейсов (новая домашка без
структурных привязок ещё не вписана в курс).

## Что проверить в будущем

### Паттерн для ловли похожих багов

**Триггер:** работа с денормализованным полем (одно и то же
значение в нескольких таблицах).

**Чек-лист:**

- [ ] Какие таблицы содержат это значение? (Найти все через
      `\d <table>` или migrations/* за фразу).
- [ ] Какая из них — **single source of truth**? (Та, в которой
      значение заполнено всегда / эксклюзивно создаётся).
- [ ] Для каждой остальной — как она получает значение?
      Trigger? Backfill? Ручная синхронизация?
- [ ] Может ли эта таблица иметь NULL в этой колонке? (Если да,
      нужна fallback-цепочка).
- [ ] В UI / коде, который использует значение — есть ли явный
      `?? fallback`? Или предполагает «всегда заполнено»?

**Если шаг 5 = «предполагает» — это бомба.** Раньше или позже
NULL прилетит (новая запись, миграция-edge, backfill пропустил),
и UI сломается.

### Рекомендуемая структура fallback-цепочки

```js
const value = primary
    ?? secondary_via_strongest_fk
    ?? tertiary_via_other_fk
    ?? quaternary_via_indirect_fk
    ?? null;  // phantom — едва ли встречается, но ловим
```

Порядок:
1. **Primary** — то, что лежит в той же строке (после backfill).
2. **Secondary** — через FK с **уникальным** связующим (как
   `external_key`). Один-в-один.
3. **Tertiary** — через FK с **слабой** связью (`week_id`, может
   быть NULL).
4. **Quaternary** — через двойной FK прыжок (`lesson → week →
   module`). Два шанса упасть в NULL по дороге.
5. **`?? null`** — fallback на «нет данных». UI должен
   корректно отрендерить (не падать), даже если все 4 источника
   пусты.

### Сигналы

- Группировка / агрегация по полю даёт неожиданно много в
  «Без / Без значения» категории = backfill / источник правды
  не покрывает большинство данных.
- Поле `column int` без `NOT NULL` constraint в схеме = автор
  таблицы заранее знал, что будут NULL'ы. Это полезный сигнал —
  ищите fallback-стратегию **до** того, как будете писать
  использование.
- Поле существует в нескольких таблицах = спросите автора, какая
  из них канонична. Если ответ «и та, и та, как удобнее» —
  смотрите, как они синхронизируются. Часто никак.

### Долгосрочно: нормализовать или server-side агрегировать

**Опция A — нормализовать.** Удалить `module_number` из всех
таблиц кроме одной. Везде ходить по FK к этой одной. Минус:
больше JOIN'ов в каждом read-запросе → дороже на read-heavy
endpoints.

**Опция B — RPC-агрегатор.** Как
`public.pvl_admin_progress_summary(p_cohort_id)` — вычисляет
все нужные join'ы и агрегации на стороне БД, возвращает
готовый jsonb. Клиенту не нужно знать про fallback-цепочку —
сервер всё сделал.

Опция B **предпочтительна** для админ-views, где сложная
аггрегация. См. PERF-001-ADMIN-API в backlog. На клиенте
fallback-цепочка должна остаться как safety-net для
пользовательских views, где RPC-агрегатор overhead.

### Тест на регресс

Не пишу автотест — fallback-цепочка работает только при наличии
реальных данных в `pvl_content_items`/`pvl_course_weeks`/
`pvl_course_lessons`. Юнит-тест с моками будет проверять «код
вызывает Map.get», а не семантику.

Лучший тест — данные. Когда появится TEST-INFRA-SETUP — фикстура
с тремя homework_items: один с `module_number=2` (proves primary),
один с NULL + valid `external_key` → content_item с
`module_number=3` (proves fallback 1), один со всеми NULL'ами
кроме `lesson_id` → lesson → week с `module_number=1` (proves
fallback 3). Регресс: если кто-то удалит ?? в цепочке — два из
трёх перестанут попадать в правильную группу.
