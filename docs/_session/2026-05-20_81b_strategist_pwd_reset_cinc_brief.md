## Сценарий (read-only)

### Шаг 1. Открыть и подготовить

1. Открыть в новой вкладке https://liga.skrebeyko.ru/
2. Открыть DevTools (Cmd+Opt+I)
3. Перейти на вкладку **Network**, поставить фильтр на
   `auth.skrebeyko.ru` (или ничего, если фильтра нет — посмотрим все)
4. Очистить список requests (clear)
5. **Не логиниться** в систему — нужна форма AuthScreen для
   незалогиненного пользователя
6. Если AuthScreen не появляется автоматически — перейти на
   `https://liga.skrebeyko.ru/login` или `/auth`

### Шаг 2. Найти кнопку

Найти на AuthScreen ссылку/кнопку **«Забыли пароль?»** (по скриншоту
вчера была под полем пароля, справа). Сделать **скриншот** этой
секции AuthScreen до клика.

### Шаг 3. Клик и наблюдение (БЕЗ ввода email сначала)

1. Кликнуть на «Забыли пароль?» **без** заполнения email поля
2. Записать что произошло:
   - Перешло на другую страницу? (какой URL?)
   - Открылась модалка / попап?
   - Появилось дополнительное поле для ввода email?
   - Toast / alert «введите email»?
   - Ничего? (noop)
3. Скриншот результата клика
4. В Network — улетел ли какой-то API request? Какой URL, method, status?

### Шаг 4. Если есть поле для ввода email — заполнить тестовым

1. Ввести **несуществующий** email: `test_pwdreset_does_not_exist@example.invalid`
   (специально несуществующий, чтобы посмотреть как backend реагирует
   на unknown email — это важная информация про security/UX, что
   написано в response, что показано юзеру)
2. Очистить Network снова
3. Нажать «Отправить» / «Восстановить» / submit-кнопку
4. Записать:
   - Какой URL улетел (полный, включая `auth.skrebeyko.ru/auth/...`)
   - Method (POST/GET)
   - Payload (что в request body)
   - Response status (200/404/501/что-то ещё)
   - Response body (полностью если короткий, или ключевые поля)
   - Что показалось пользователю в UI (toast / новая страница /
     ничего / generic «Неверные данные» как мы видели вчера)
5. Скриншот UI + DevTools Network panel с раскрытым request'ом

### Шаг 5. Если эндпоинт реально работает (200 OK) — НЕ повторять с реальным email

Если на Шаге 4 backend ответил 200 и явно отправляет email — **STOP,
не пытайся с реальным email**, чтобы не засорить inbox реального
пользователя. Зафиксируй что endpoint работает и пускай codeexec в
БД проверит создалась ли token-row.

Если 404 / 501 / 500 — это значит endpoint частично или полностью не
готов, тогда мы сами увидим что строить.

## Что записать в отчёт

Файл: `docs/_session/2026-05-20_82b_cinc_pwd_reset_recon.md`

Структура (короткая, ~30-50 строк):
1. **Скриншот AuthScreen** с кнопкой «Забыли пароль?» до клика
2. **Что делает кнопка** при клике без email — словами + скриншот
3. **Что делает submit** с несуществующим email — словами + скриншот UI +
   данные из Network panel (URL, method, status, body)
4. **Какой URL endpoint'а** (мы заберём для имплементации/тестов)
5. **Краткий вывод:** «UI готов и работает / UI готов но backend не
   готов / UI заглушка / ничего нет»

## Что НЕ делать

- ❌ Не залогиниваться существующим email (Ольгиным, чужим) — должны
  быть в незалогиненном состоянии
- ❌ Не запрашивать reset на реальный email — даже Ольгин — чтобы не
  засорить inbox и не invalidate её текущий JWT нечаянно
- ❌ Не пытаться сделать сам reset через сгенерированную ссылку
  (если случайно появится в response/email) — это уже apply-действие
- ❌ Не показывать в отчёте полные cookie/JWT (если они каким-то образом
  попадут в snapshot Network) — обрезать `Authorization: Bearer ...`
  и `Cookie: ...`
