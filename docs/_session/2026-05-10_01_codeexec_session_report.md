### 2.2 Реальные баги, обнаруженные на проде

**Баг 1 — beзвучный disable** (`13c34a3`).
Симптом: кнопка не реагирует на клик. Причина: `disabled={modules.length === 0}` + молчаливый `.catch(() => [])` в загрузке. Фикс: показать ошибку явно, не давить кнопку при пустых модулях.

**Баг 2 — `module_number = NULL` в `pvl_homework_items`** (`0b50fbf`, `4eea3d8`, `6c2a2c1`).
Симптом: dropdown пустой, скачанный MD без секций. Причина: `phase 25 backfill` заполняет `module_number` только для записей с title по regex «модул[ьюяе] N». Этот regex почти ничего не покрыл. На проде из 22 ДЗ — `module_number = NULL` для большинства.

Финальный фикс — цепочка из четырёх фолбэков в `effectiveModuleNumber`:
1. `pvl_content_items.module_number` через `external_key === task-ci-{ci.id}` ← главный.
2. `pvl_homework_items.module_number` напрямую.
3. `pvl_course_weeks.module_number` через `week_id`.
4. `pvl_course_lessons.module_number` через `lesson_id`.

**Баг 3 — Vite hash collision** (`4fd8a4d`).
Симптом: bundle на проде вроде новый (`index-DvDJAQqT.js`), но в браузере висит старый код. Причина: два разных билда получили один и тот же contenthash, plus `Cache-Control: public, immutable, max-age=31536000` от Vite не даёт браузеру перезагрузить. Фикс: внести минорную правку в код для гарантированной смены hash. Урок отдельно записать.

**Баг 4 — race condition в dropdown'е** (`980fd45`).
Симптом: «модули то показываются, то нет». Причина: dropdown открывался до того как Promise.all завершался, useMemo возвращал `[]`, и dropdown зависал в этом состоянии. Фикс: явный state `reportDataReady`, кнопки disabled до готовности данных.

**Баг 5 — N последовательных запросов** (`193c999`).
Симптом: «Готовлю архив… 0/14» зависает на минуты, ZIP не скачивается. Причина: `pvl_student_homework_submissions` отвечает на проде **~190 секунд на одну студентку**. Я делал последовательно `for ... await` 14 запросов = 45 минут. Фикс: новый `listStudentHomeworkSubmissionsBulk(studentIds)` через `student_id=in.(...)` — один запрос вместо 14. Прогресс перестал быть пер-студент (теперь 0 → total/2 → total).

### 2.3 Подтверждено на проде после `193c999`

Smoke от Ольги: «вроде всё быстро и хорошо прошло». ZIP скачивается, модули в dropdown'е видны, отдельный MD по студентке открывается за секунды.

## 3. Что мы заметили — для протокола

### 3.1 PostgREST на проде очень медленный для read-heavy admin-фич

Из smoke-отчёта Claude-в-браузере 2026-05-09:
- `pvl_content_items`: 176 секунд.
- `pvl_student_homework_submissions` (per студент): 192 секунды.
- `knowledge_base`: 43 секунды.

Это блокер для интерактивных дашбордов. Скорее всего связка тяжёлых RLS политик + jsonb payload без индексов. Не лечится фронтом — нужен либо профайл RLS, либо специальные RPC-агрегаторы под админский use-case (как `pvl_admin_progress_summary` для phase 25). Пока живём с долгой первой загрузкой данных курса (~3 минуты на «Прогресс ПВЛ»).

### 3.2 Связь homework_item ↔ модуль курса не нормализована

Сейчас «номер модуля» лежит в 4 разных таблицах с разной достоверностью. Долгосрочно — перенести на сервер: либо `pvl_admin_progress_summary` расширить, чтобы возвращало homework_items с резолвленным module, либо новый RPC `pvl_admin_homework_inventory(cohort_id)`. Тогда фронт перестанет тащить `pvl_content_items` напрямую.

### 3.3 Видимость пользователя в саду — client-only

`hiddenGardenUserIds` в `localStorage` одного админа. Ольга справедливо заметила: «Если я как админ скрываю — он должен быть скрыт у всех». Это MVP-костыль, нужен серверный флаг.

### 3.4 Bundle на 1.33 MB

Main bundle везёт `AdminPanel`, `BuilderView` (с jspdf+html2canvas), `MeetingsView`, `dataService.js` — всё статически. У не-админских ролей выкатывается лишний JS. План оптимизации записан.

### 3.5 Vite immutable cache — слепая зона

Во время дебага я думал что фикс уехал на прод, а у Ольги в браузере был тот же URL `index-XXX.js` с **другим контентом** (hash collision). Стандартный hard reload не пробивает `immutable, max-age=31536000`. Будем помнить: при странных «починил, но не работает» — сравнить hash bundle на проде с локальным через `curl + grep`, и если совпали при разном коде — насильно бустить hash.

## 4. Followups в очереди

### 4.1 Серверная видимость пользователя в саду

**Когда:** ближайшая задача после стабилизации.

**Что:**
- Миграция: `ALTER TABLE profiles ADD COLUMN is_hidden_in_garden boolean NOT NULL DEFAULT false`.
- RLS: SELECT всем authenticated, UPDATE только админу.
- Backfill: `UPDATE profiles SET is_hidden_in_garden = true WHERE id IN (...)` — Ольга экспортирует свой текущий localStorage и засеется.
- API: `api.toggleUserVisibilityInGarden(id)`.
- Frontend: убрать `localStorage` костыль, всё переключить на серверный флаг + `users` refetch.
- Бонус-таска: кнопка «удалить из курса» на дашборде «Прогресс ПВЛ» — для случаев когда тестовый юзер попал в `pvl_students` и его нужно вычистить (Настин фиксик-кейс).

### 4.2 Новый критерий «в ритме» + приём задним числом

**Когда:** ждём фидбэк менторов о работе с гугл-доками.

**Что (черновик):**
- В RPC `pvl_admin_progress_summary` пересмотреть `state_line`: «в ритме» = `hw_accepted >= ожидаемое-к-сегодня`, где «ожидаемое» считается по `pvl_course_weeks.starts_at/ends_at` для текущей даты.
- UI: кнопка «принять задним числом» в карточке студентки → создаёт submission с `status=accepted`, `accepted_at = выбранная-дата`. Для гугл-доков и других внеплатформенных сдач.
- Решить: где хранить ссылку на гугл-док (поле `payload.external_link`?) или просто комментарий ментора со ссылкой.

### 4.3 Bundle optimization

**Когда:** после закрытия 4.1, чтобы не смешивать функциональные правки с рефакторингом.

**Где:** [`plans/2026-05-09-bundle-optimization.md`](../../plans/2026-05-09-bundle-optimization.md).

**Кратко:** lazy AdminPanel + BuilderView (выкинут jspdf/html2canvas из main), потом остальные view'ы, потом manualChunks. Цель — main bundle ≤ 500 KB / ≤ 150 KB gzip.
