## Шаг 1 — можно ли тянуть Prodamus API напрямую? **НЕТ.**

- Prodamus REST API (`/rest/{method}/`) отдаёт **только управление подписками**: `setActivity`,
  `setSubscriptionDiscount`, `setSubscriptionPaymentDate`. **Метода «список платежей/выгрузка транзакций
  нет.** Экспорт платежей — только вручную из ЛК (раздел «Список оплат», фильтр/скачивание, до 12 мес).
- push-server Prodamus **только принимает** вебхук (верификация подписи `PRODAMUS_SECRET_KEY`), исходящего
  API-клиента нет; в env нет рабочего ключа для выгрузки.
- БД **не авторитетна** (recon 214: 13+ плательщиков с пустым `paid_until` — вебхук не долетает).

**→ Источник правды = ручная CSV-выгрузка из ЛК Prodamus.** Использовал уже лежащую на диске
`paylist (1).csv` (от 2026-07-09 12:42). **Подтверди, что это твоя актуальная 2-мес выгрузка** — тогда
числа ниже валидны; если она устарела, дай свежую и перегоню.

## Расчёт `paid_until`
`paid_until = (последний Лига-платёж) + 1 мес`. Идентичность плательщика — по `tg_user_id`, иначе email,
иначе тел/имя; несколько платежей одного человека слиты (берём последний).

## ⚠️ ПОЛИТИКА ЗАПИСИ (критично для будущего массового UPDATE)
**`paid_until = GREATEST(текущий DB, расчётный)` — НИКОГДА не уменьшать.** Есть случаи, где DB > расчётного:
- **Мария Романова:** DB `2026-08-08`, расчёт `2026-07-09` → слепая запись **сократила бы** ей доступ на месяц.
- Множество строк с `*` — расчёт на 1 день меньше DB (webhook стекал иначе). Тоже не уменьшать.
Реальные изменения = только строки с пустым DB (`—`), поднятые до реальной даты.
