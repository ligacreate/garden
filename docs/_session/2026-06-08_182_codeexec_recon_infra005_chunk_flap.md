# RECON INFRA-005 — SW/cache chunk-flap (повторяющиеся ChunkLoadError)

**Дата:** 2026-06-08
**Тип:** read-only разведка, фикс предложен, не применён
**Сервер:** liga.skrebeyko.ru · **Репо:** garden
**Симптом:** через дни после деплоя у разных юзеров — ChunkLoadError → auto-reload, повторяется.

---

## TL;DR — вердикт

**Корень: 404 на сервере.** Деплой `dangerous-clean-slate: true` стирает весь
каталог при каждой выкладке → старые хеш-чанки удаляются. Долгоживущие вкладки
(SPA, без полной перезагрузки днями) держат в памяти старый entry-чанк, который
лениво (`React.lazy`/`import()`) тянет view-чанки по **старому** хешу. После
деплоя этот хеш → 404 → ChunkLoadError при клиентской навигации на ленивый роут
→ ErrorBoundary делает auto-reload.

**НЕ корень:** index.html кэшируется (опровергнуто), SW держит старый index
(опровергнуто). Заголовки кэша настроены **корректно**.

**Минимальный фикс:** перестать удалять старые чанки при деплое (additive-выкладка
с ретенцией). Плюс вторичный баг: loop-guard в ErrorBoundary не сбрасывается →
при втором деплое в одной сессии юзер видит красный экран ошибки.

## 1. Живы ли ассеты на сервере (HEAD)

| Ассет | Статус |
|---|---|
| `/assets/LeaderPageView-B9Lupfs6.js` | **404** |
| `/assets/MeetingsView-B9okXXi_.js` | **404** |
| `/assets/CourseLibraryView-ChSgt2Ze.js` | **404** |
| `/assets/index-qW9hXWy2.js` | **404** |

Текущий серверный `index.html` ссылается на **новый** entry:
```
src="/assets/index-C61HUHpU.js"
href="/assets/index-DUfUHP98.css"
```
На старый `index-qW9hXWy2.js` ссылок **нет**. → Это нормальный новый билд; все
старые чанки физически удалены с сервера (отсюда 404).

## 2. Заголовки кэша (curl -I) — корректны

| Ресурс | Cache-Control | Вердикт |
|---|---|---|
| `index.html` / `/` | `no-cache` | ✅ верно (браузер ревалидирует) |
| `index-C61HUHpU.js` (живой) | `public, immutable, max-age=31536000` | ✅ верно |
| `index-DUfUHP98.css` (живой) | `public, immutable, max-age=31536000` | ✅ верно |
| `sw.js` | `max-age=86400` | ⚠️ минор (см. ниже) |

Заголовки **не** являются корнем: index.html свежий, хеш-ассеты immutable —
это ровно та конфигурация, что нужна.

## 3. Service Worker (`public/sw.js`) — не держит старый index

- `install` → `skipWaiting()` (мгновенная активация).
- `activate` → удаляет **ВСЕ** caches (`caches.keys().then(delete all)`) +
  `clients.claim()`.
- `fetch` → только для `mode === 'navigate'`: `fetch(req, {cache:'reload'})`,
  на ошибку → `Response.error()`. **Ничего не precache'ит, ни во что не падает
  как в кэш.**

→ SW network-first для навигации и вообще не хранит index.html. Версия
«SW держит старый index» **опровергнута**: стейл-индекса взяться неоткуда.

## Дымящийся ствол — деплой

`.github/workflows/deploy.yml`, шаг «Deploy via FTP»:
```yaml
uses: SamKirkland/FTP-Deploy-Action@v4.3.5
with:
  ...
  dangerous-clean-slate: true   # ← стирает весь server-dir и заливает заново
```
Каждый деплой полностью очищает каталог и кладёт только текущий билд → все
прошлые хеш-чанки исчезают мгновенно. (Косвенное подтверждение, что команда в
курсе тотального сноса — комментарий в шаге «Prepare deploy bundle» про
`.htaccess`, который из-за clean-slate приходится перезаливать каждый раз.)

## Механика бага (end-to-end)

1. Юзер держит вкладку открытой днями (SPA, навигация клиентская, без F5).
2. Загруженный `index-<старый>.js` лениво импортит view-чанки по старым хешам.
3. Новый деплой → `dangerous-clean-slate` сносит каталог, заливает новые хеши.
   Старые view-чанки → 404.
4. Юзер кликает на ленивый роут (Meetings / CourseLibrary / LeaderPage) →
   `import('…-<старый-хеш>.js')` → 404 → ChunkLoadError.
5. `ErrorBoundary.componentDidCatch` ловит → `window.location.reload()` →
   тянет свежий `index.html` (`no-cache`) → новые чанки → снова работает.

Поэтому это «через дни после деплоя» и «у разных юзеров» — у каждого свой момент
первой навигации на ленивый роут после выкладки.

## Вторичный латентный баг — loop-guard в ErrorBoundary

`components/ErrorBoundary.jsx`:
```js
if (!sessionStorage.getItem('garden_chunk_reloaded')) {
  sessionStorage.setItem('garden_chunk_reloaded', String(Date.now()));
  window.location.reload();
  return;
}
```
Флаг ставится **один раз за сессию вкладки и никогда не сбрасывается**.
`sessionStorage` переживает reload в той же вкладке. → Если в этой же долгой
сессии случится **второй** деплой и юзер откроет ещё не загруженный ленивый
чанк, guard уже стоит → reload **не** сработает → юзер упрётся в красный экран
«Что-то пошло не так». Защита от loop работает только против ПЕРВОЙ ошибки.

## Предлагаемый минимальный фикс

### Главный (owner layer = deploy/сервер): не удалять старые чанки

Цель — старые хеш-чанки переживают деплой, чтобы стейл-вкладки догрузили свои
старые модули и дожили до естественной перезагрузки. Хеш-имена не конфликтуют —
старое и новое спокойно сосуществуют.

Варианты реализации (по возрастанию надёжности):

- **A. Additive-выкладка `assets/`.** Перестать сносить каталог; заливать новое,
  старые `assets/*` не трогать. Учесть: голое снятие `dangerous-clean-slate`
  у FTP-Deploy-Action оставляет diff-синхронизацию, которая всё равно удалит
  файлы, исчезнувшие из билда. Значит — нужно либо `exclude` на `assets/**` от
  удаления, либо before-build подмешать в deploy-bundle текущие live-ассеты
  (скачать `assets/` с сервера → слить со свежими → выложить). Тогда ничего
  никогда не пропадает.
- **Ретенция.** Каталог `assets/` будет расти → периодическая чистка
  (cron/ручная): удалять ассеты старше N дней или оставлять M последних сборок.
  Контролируемый рост вместо мгновенного 404.

Этот фикс **устраняет** ChunkLoadError как класс: 404 не возникает → reload не
нужен → пропадает раздражающий «рандомный» auto-reload.

### Дополнительный (child layer): починить guard

Сделать `garden_chunk_reloaded` самосбрасывающимся — хранить timestamp и
игнорировать его, если прошло > N минут (или хранить версию билда и сбрасывать
при смене). Тогда auto-reload корректно отработает на каждый новый деплой, а не
только на первый за сессию. Это страховка, не замена главному фиксу.

### Минор-гигиена

`sw.js` отдаётся с `Cache-Control: max-age=86400` — скрипт SW кэшируется на
сутки. Современные браузеры всё равно ревалидируют SW (cap 24ч) и обходят HTTP-
кэш при обновлении, так что это не корень. Но при изменении логики SW обновление
может запаздывать до суток — разумно отдавать `sw.js` как `no-cache` (в
`.htaccess`/nginx). Низкий приоритет.

---

## Что НЕ нужно делать

- Не трогать заголовки index.html / хеш-ассетов — они уже верные.
- Не «чинить» SW под стейл-индекс — SW не виноват, network-first уже корректен.
- Не оставлять как есть, полагаясь только на auto-reload — это лечение симптома
  в дочернем слое; корень в слое деплоя/сервера (404), там и чинить.

## Карта затронутых слоёв (для фикса)

- **Деплой:** `.github/workflows/deploy.yml` — `dangerous-clean-slate` + ретенция.
- **Сервер:** `.htaccess`/nginx — (минор) `no-cache` на `sw.js`.
- **Клиент:** `components/ErrorBoundary.jsx` — самосброс loop-guard.
