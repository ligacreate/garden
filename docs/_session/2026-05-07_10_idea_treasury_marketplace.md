## Полный текст ТЗ от Ольги (без изменений)

### КОНТЕКСТ

В Garden (liga.skrebeyko.ru) нужно добавить новый раздел
«Сокровищница» — общая бесплатная база практик Лиги. Раньше
такого места не было: «Практики» — это личная коллекция ведущей,
«Библиотека» занята обучающими материалами и курсами, «Магазин» —
скидки и продукция издательства. Места для «общих рабочих
инструментов от ведущих для ведущих» нет.

Параллельно нужно заложить архитектуру под будущий маркетплейс
авторских платных сценариев и воркбуков (UI делать сейчас не надо,
только модель данных).

### ЗАДАЧА 2. Новый раздел «Сокровищница»

Новый пункт меню «Сокровищница». Видна всем ведущим Лиги.

Сейчас внутри: только практики Лиги.
В будущем: добавятся бесплатные сценарии Лиги (реализация позже,
архитектуру предусмотреть).

#### 2.1. Поток публикации в Сокровищницу

Любая ведущая может опубликовать свою практику в Сокровищницу.
Поток:

1. В разделе «Мои практики» у каждой карточки кнопка
   «Предложить в Сокровищницу».
2. По клику открывается форма публикации с обязательными
   полями (см. модель ниже). Если в личной карточке поля не
   заполнены — заставить заполнить здесь.
3. После сохранения карточка получает статус
   `moderation_status = «на ревью»`.
4. Куратор (роль) видит очередь модерации в админке.
   Может «Одобрить» или «Отклонить с комментарием».
5. При одобрении: карточка публикуется в Сокровищнице
   (`is_published = true`), автору начисляются семена
   (см. ЗАДАЧА 3).

#### 2.2. Поток «форк» — добавить себе в коллекцию

На каждой карточке в Сокровищнице кнопка «Добавить в мою
коллекцию». По клику у ведущей в «Мои практики» появляется
копия с пометкой «из Сокровищницы, автор [имя]».

Атрибуция автора сохраняется навсегда. Ведущая может
адаптировать копию под себя (поменять текст), но автор
исходника остаётся виден.

**ВОПРОС от Ольги:** что технически лучше — копировать запись или
хранить связь по ссылке? Что произойдёт, если автор обновит
оригинал — должна ли копия обновиться? Предложи решение
с обоснованием.

#### 2.3. Поиск и фильтры в Сокровищнице

Сейчас на странице «Практики» фильтры скудные — поиск,
«Любое время», теги «Все/Вдохновение». В Сокровищнице
нужно полноценно:

- Полнотекстовый поиск (по названию, тексту, тегам, автору)
- Фильтр по тегам (мультивыбор с автодополнением)
- Фильтр по теме
- Фильтр по длительности (диапазоны: до 10 / 10–20 / 20–40 / 40+ мин)
- Фильтр по типу практики
- Фильтр по нагрузке (лёгкая / средняя / тяжёлая)
- Фильтр по контексту применения (разогрев / основная /
  рефлексия / выход)
- Фильтр по формату (онлайн / офлайн / оба)
- Сортировка: новизна / популярность (по числу форков) /
  алфавит / автор

### ЗАДАЧА 4. Расширение модели Practice

Нужно добавить поля в модель Practice. Они нужны для
Сокровищницы (фильтры, поиск, авторство), но для личной

#### ОБЯЗАТЕЛЬНЫЕ ПРИ ПУБЛИКАЦИИ В СОКРОВИЩНИЦУ:

- `author_id` — ссылка на пользователя-ведущую

- `usage_context` — массив enum: разогрев / основная /
  рефлексия / выход

#### ОПЦИОНАЛЬНЫЕ:

- `audience_readiness` — enum: холодная / тёплая / опытная
- `materials` — text (что нужно физически)

- `reflection_questions` — text (вопросы для рефлексивного

#### ПОД БУДУЩИЙ МАРКЕТПЛЕЙС (заложить, UI пока нет):

- `price` — decimal (рубли)

  из Сокровищницы — ссылка на оригинал)

### ЗАДАЧА 5. Заложить архитектуру маркетплейса (без UI)

Контекст: в будущем в разделе «Магазин» появится зона
«Авторское» — ведущие Лиги продают друг другу сценарии
и воркбуки. Деньги идут мимо платформы.

UI маркетплейса сейчас НЕ делать. Заложить только модель
данных + механику доступа.

#### 5.1. Механика доступа (ручная)

Поскольку деньги вне платформы, доступ открывает сам автор:

1. Покупатель на карточке нажимает «Связаться с автором» —
   видит контакт.
2. Они договариваются и платят вне платформы.
3. Автор в своём интерфейсе нажимает «Открыть доступ» →
   выбирает покупателя из списка ведущих → у того в
   «Мои сценарии/Практики» появляется копия с атрибуцией
   «куплено у [автор], дата».

Промокоды одноразового доступа — НЕ делаем. Продажи будут
штучные, не сотнями.

#### 5.2. Таблица `purchase_log`

Поля:

- `author_id`
- `buyer_id`
- `item_id` (scenario / practice)
- `item_type` (scenario / practice)
- `price_at_sale` (фиксируем цену на момент сделки)
- `created_at`

#### 5.3. Раздел «Мои продажи» в профиле ведущей

В профиле автора — экран со списком: что продано, кому,
когда, на какую сумму. Простая таблица.

## Короткие первичные комментарии стратега (без полного аудита)

Это **не план** — это base-instinct ответ для пометки. Полный план
после аудита кода + DB.

### По вопросу 2.2 (форк — копия или связь?)

Рекомендация: **hybrid — копия с FK `forked_from` + `forked_from_version_at`**.

- Копия → ведущая может адаптировать под себя без боязни «автор обновит и
  мои правки слетят».
- FK → атрибуция сохраняется навсегда (на UI всегда «автор: [имя]»).
- `forked_from_version_at` (timestamp оригинала на момент форка) → если
  автор обновит оригинал, у форка появляется индикатор «Доступна
  обновлённая версия от [автор]». Ведущая сама решает: смержить или
  оставить свою адаптацию.

Это уважение к обеим сторонам.

### По вопросу 3 (где логика семян)

**Без recon'a:** в Garden есть таблица seed-related (упоминается
`increment_user_seeds` в нашем DB-recon-сегодняшнего-дня — есть RPC).
Триггер «после одобрения публикации» можно сделать как:
- option A: вызов RPC `increment_user_seeds(user_id, amount)` из
  серверного кода после `UPDATE moderation_status='approved'`.
- option B: триггер в БД на `AFTER UPDATE OF moderation_status` с
  условием `WHEN NEW.moderation_status='approved' AND NEW.seeds_awarded=false`.

Option B — чище (атомарная транзакция, нельзя «забыть» вызвать). На
аудите проверим, как сейчас RPC вызывается.

### По «Сокровищница: отдельная таблица или флаг»

Рекомендация: **флаг (`is_published` + `moderation_status`) на существующей
Practice.**

- Это **режим публикации** той же сущности, не новая модель.
- Форк копирует Practice as-is (поля совпадают).
- RLS для Сокровищницы строится на `WHERE is_published=true AND moderation_status='approved'`.
- Отдельная таблица оправдана только если появятся принципиально
  другие поля. Сейчас их нет.

### По полиморфности (практики + сценарии)

Рекомендация: **одинаковые поля публикации в обеих таблицах, но не
общая `publishables` таблица.**

- Поля `is_published`, `moderation_status`, `published_at`,
  `seeds_awarded`, `forked_from` повторяются в `practices` и в
  `scenarios` с одинаковыми именами.
- Полиморфная модерационная очередь и Сокровищница собирают через
  `UNION ALL` view или RPC.
- Это проще для миграции (Practice/Scenario не меняем структурно),
  для тестирования и для будущего расширения (если появится третий
  тип — добавить такую же группу полей).
