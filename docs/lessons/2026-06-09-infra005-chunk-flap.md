# INFRA-005: ChunkLoadError у долгих вкладок — clean-slate сносил старые хеш-чанки (корневой фикс)

**Дата:** 2026-06-09.
**Связанные коммиты:** `6f1d199` (v1, откатан `a5acb77`), `e989474` (v2), `0736b75` (v3 — рабочий).
**Связанные сессии:** [_182 recon](../_session/2026-06-08_182_codeexec_recon_infra005_chunk_flap.md), [_184 diff v1](../_session/2026-06-09_184_codeexec_infra005_deploy_fix_diff.md), [_185 diff v2](../_session/2026-06-09_185_codeexec_infra005_v2_prefetch_diff.md), [_186 diff v3](../_session/2026-06-09_186_codeexec_infra005_v3_envpassword_diff.md).
**Связанные тикеты:** [[INFRA-005-SW-CACHE]] (симптомный фикс 2026-05-11 через ErrorBoundary auto-reload — это лечило проявление; здесь закрыт корень), [[INFRA-005-ASSETS-RETENTION]] (новый P3 — побочка фикса).

## Симптом

Через дни после деплоя у разных юзеров — `ChunkLoadError` (`Failed to fetch
dynamically imported module …`), ErrorBoundary делал auto-reload. «Рандомно» и
«у разных»: у каждого свой момент первой клиентской навигации на ленивый роут
после очередной выкладки.

## Корневая причина

Слой **деплоя**, не клиента и не SW. `.github/workflows/deploy.yml` заливал с
`dangerous-clean-slate: true` → каждый деплой **сносил весь server-dir** и
заливал только текущий билд → все прошлые хеш-чанки исчезали мгновенно.
Долгоживущие SPA-вкладки держат в памяти старый entry-чанк, который лениво
(`import()`) тянет view-чанки по **старым** хешам → после деплоя те 404 →
ChunkLoadError. Заголовки кэша и SW были корректны (опровергнуто в recon _182).

## Почему так получилось

clean-slate — простой способ «чистого» деплоя без diff-синхронизации, но он
несовместим с immutable-хешированными ассетами: старое и новое **обязаны
сосуществовать**, пока живут старые вкладки. Хеш-имена не конфликтуют — снос был
не нужен, а вреден.

## Как починили (owner layer = деплой)

Финал (v3, `0736b75`): **префетч + мерж перед clean-slate-выкладкой**. Новый шаг
ПЕРЕД SamKirkland скачивает текущие серверные `assets/` (`lftp mirror`) и мержит
в `deploy/assets/` без перезатирания свежесобранных (`cp -rn`). clean-slate
остаётся, но теперь bundle уже содержит и новые, и все старые чанки → старое
переживает выкладку. Шаг non-fatal (`continue-on-error` + `set +e`): сбой
префетча → деплой как раньше, не хуже. Проверка: лог печатает «файлов после
мержа: N», N>свежего-билда; старый чанк прошлой выкладки отдаёт 200 после новой.

Дочерний слой (страховка, уже был с _184): `components/ErrorBoundary.jsx` —
loop-guard `garden_chunk_reloaded` теперь самосбрасывается по времени (окно 30с),
чтобы второй деплой в одной долгой сессии тоже корректно перезагружал.

## Грабли по пути (v1→v3) — главное к запоминанию

1. **`lftp mirror` additive по умолчанию — флага `--no-delete` НЕ существует.**
   v1 (`6f1d199`) добавлял `mirror -R --no-delete …` → `mirror: unrecognized
   option '--no-delete'` → шаг упал. Удаление у `mirror`/`mirror -R` включается
   **только** через `--delete`/`-e`; без него ничего не удаляется. Не выдумывать
   «защитные» флаги — проверять `lftp -c "help mirror"`.

2. **v1 оставил прод в полу-сломанном состоянии → правило про порядок uploadّа.**
   v1 снял clean-slate и добавил `exclude: assets/**` у action: action успел
   залить новый `index.html` (новые хеши), а отдельный lftp-аплоад чанков упал →
   index ссылается на чанки, которых нет → риск 404 на entry. **Урок:** если
   index и его чанки доставляются разными шагами, провал второго оставляет прод
   битым. Безопаснее доставлять index и чанки **одним** атомарным bundle'ом
   (что и делает v3: всё уезжает одной clean-slate-выкладкой). При красном
   workflow по deploy.yml — откат deploy.yml (сделали: `a5acb77` → прод
   восстановлен clean-slate'ом за один передеплой).

3. **FTP-пароль в CI — только через env (`--env-password`/`LFTP_PASSWORD`), НЕ
   в строке команды.** v2 (`e989474`) передавал пароль как `user "$PASS"` внутри
   двойно-кавыченной `lftp -e "…"` → `530 Login incorrect` (спецсимвол в пароле
   манглился при подстановке bash→lftp; те же секреты у SamKirkland работали).
   v3 (`0736b75`): `export LFTP_PASSWORD="$FTP_PASSWORD"; lftp --env-password -u
   "$USER" …` → пароль не попадает в строку → устойчив к любым спецсимволам.
   То же для имени: `-u "$USER"` (без `-u user,pass` — запятая в пароле сломала
   бы и это).

## Что проверить в будущем

- **При любой схеме «не удалять старое» на FTP:** убедиться, что инструмент
  именно additive (lftp mirror — да; clean-slate — нет). Проверять не догадкой,
  а `help <cmd>`.
- **Шаг с FTP/lftp в CI:** креды только через env, не инлайном; шаг,
  доставляющий ретеншн, делать non-fatal, чтобы не валить основной деплой.
- **Сигнал, что фикс работает:** старый хеш-чанк (entry прошлого билда) отдаёт
  **200 после новой выкладки**. Если 404 — префетч/мерж не сработал (смотреть
  «файлов после мержа: N» в логе шага и строки `Login`/`mirror:`).
- **Побочка:** `assets/` растёт каждый деплой → нужна ретенция (см.
  [[INFRA-005-ASSETS-RETENTION]]).
