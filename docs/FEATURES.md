## Легенда статусов
- 🟢 STABLE — работает, в проде, активно используется
- 🟡 BUGGY — работает, но с известными проблемами
- 🔵 WIP — в разработке, не готово
- ⚪ LEGACY — работает, но устарело, может быть удалено

## 1. Авторизация и управление аккаунтом

### Регистрация
- **Статус:** 🟢 STABLE
- **Кто использует:** все новые пользователи
- **Что делает:** создаёт аккаунт по email/паролю/ДР, рассчитывает «дерево друидов» по дате рождения, выдаёт роль `applicant`, ставит случайные `x/y` для карты.
- **Где в UI:** [AuthScreen.jsx](../views/AuthScreen.jsx) → шаги «Welcome → форма → расчёт дерева → подтверждение».
- **Главные файлы:** [views/AuthScreen.jsx](../views/AuthScreen.jsx), [services/dataService.js](../services/dataService.js#L1149), [utils/druidHoroscope.js](../utils/druidHoroscope.js).
- **Сущности БД:** `profiles`.
- **Внешние сервисы:** Supabase Auth.
- **Известные проблемы:** ошибки регистрации показываются через `alert()`. Сообщение «найдите письмо подтверждения» подразумевает email-confirmation, но flow не проверен.

### Вход (login)
- **Статус:** 🟢 STABLE
- **Кто использует:** все.
- **Что делает:** логин по email/паролю, сохраняет JWT в `localStorage` под `garden_auth_token`.
- **Где в UI:** [AuthScreen.jsx](../views/AuthScreen.jsx) → форма «Войти».
- **Главные файлы:** [services/dataService.js:1110](../services/dataService.js#L1110), [App.jsx:136](../App.jsx#L136).
- **Сущности БД:** `profiles`.
- **Внешние сервисы:** Supabase Auth.
- **Известные проблемы:** при некорректных данных — `alert()`, нет inline-ошибок.

### Восстановление пароля
- **Статус:** 🟡 BUGGY
- **Кто использует:** все.
- **Что делает:** запрос ссылки на email + сброс по токену из URL `?token=...`.
- **Где в UI:** [AuthScreen.jsx](../views/AuthScreen.jsx) → «Забыли пароль» / `/?token=...`.
- **Главные файлы:** [services/dataService.js:1185](../services/dataService.js#L1185), [App.jsx:184](../App.jsx#L184).
- **Сущности БД:** `profiles`.
- **Внешние сервисы:** Supabase Auth.
- **Известные проблемы:** UI и копирайт «Войдите снова» не объясняет, что письмо может быть в спаме; ошибки — через `alert()`.

### Выход из аккаунта
- **Статус:** 🟢 STABLE
- **Кто использует:** все.
- **Что делает:** очищает JWT и состояние, возвращает на `AuthScreen`.
- **Где в UI:** Sidebar → «Выйти», ProfileView → «Выход».
- **Главные файлы:** [App.jsx:196](../App.jsx#L196), [services/dataService.js:1247](../services/dataService.js#L1247).
- **Сущности БД:** —
- **Внешние сервисы:** Supabase Auth (revoke).

### Авто-проверка сессии (subscription guard)
- **Статус:** 🟡 BUGGY
- **Кто использует:** все авторизованные.
- **Что делает:** каждые 60 сек запрашивает `getCurrentUser`; если статус доступа сменился на `paused_*` — кикает на `SubscriptionExpiredScreen`.
- **Где в UI:** фоновый `setInterval` в [App.jsx:115](../App.jsx#L115).
- **Главные файлы:** [App.jsx:115](../App.jsx#L115), [services/dataService.js:1251](../services/dataService.js#L1251).
- **Сущности БД:** `profiles.access_status`, `session_version`.
- **Внешние сервисы:** PostgREST.
- **Известные проблемы:** при потере сети тихо проглатывает ошибки; нет визуального индикатора «проверяем доступ»; `session_version` инвалидация — единственный механизм logout-all-devices.

## 🤔 Странности и наблюдения

1. **Все «коммуникации» — только админу.** Чат и канал в `UserApp` рендерятся только при `isAdmin === true`. Если идея — общая комната ведущих, то все остальные роли её не видят. Если идея — административная переписка, то она в одном пространстве с публичным каналом — можно перепутать.
2. **Семя-механика разбросана.** Правила начисления (`+5`, `+25`, `+80`, `+100`, `+150`, `+30`, `+10`, `+20`) встречаются в 8 обработчиках `UserApp.jsx`. Изменение баланса требует правок в 8 местах.
3. **Дублирование «события»/«встречи».** `meetings` и `events` живут параллельно: создаёшь встречу — триггер кладёт строку в `events`. При правке через AdminPanel/«События» можно перезатереть состояние, которое потом восстановит триггер.
4. **`MentorDashboardView` живёт сам по себе** — не импортируется ни в `UserApp`, ни в `AdminPanel`, ни в `PvlPrototypeApp`. Похоже на ранний эскиз, который потом переехал в ПВЛ.
5. **Prodamus webhook есть, UI самообслуживания — нет.** Бот для продления — внешняя система; пользователь не видит ни даты следующего списания, ни возможности отменить. На этапе монетизации это критично.
6. **Дашбордная новостная лента vs «Канал».** Один и тот же массив `news` показывается в дашборде каждому, но «канал» в `Communications` — только админу. Двойственность.
7. **«День рождения»-пуш отсутствует.** Auto-новость рендерится только в дашборде и только если пользователь зашёл; ни push, ни email-открытка не отправляются.
8. **`MarketView` пуст**, но место под него зарезервировано — возможно, продакт планировал магазин курсов/мерча и не довёл.
9. **Профиль удаляется = просто logout.** Юридический риск: в РФ/ЕС пользователь имеет право требовать удаления данных.
10. **Сообщения чата кэшируются в localStorage**, что при правке/удалении из другой вкладки даёт устаревший локальный кэш до следующего pull.
