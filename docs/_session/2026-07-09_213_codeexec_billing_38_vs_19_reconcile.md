## ⚠️ ГЛАВНЫЙ ВЫВОД

**«19 active prodamus» ≠ «19 платящих активных».** Реально с валидным будущим `paid_until` — **только 16** (3 из 19 уже истекли → `paused_expired`). А **24 активных профиля платящих ролей (leader/mentor/intern) НЕ имеют вообще никакой billing-записи** (`paid_until=NULL`, `last_prodamus_event=NULL`, `auto_pause_exempt=false`) — но у всех активный доступ и почти у всех привязан Telegram. **Это и есть «хвост»: они платят мимо платформы (TargetHunter) или перенесены вручную.** Удаление/пауза по prodamus-данным выкосит 24 легитимных участника.

Плюс: **`auto_pause_exempt=true` — 0 профилей.** Бартерные/льготные НЕ помечены в БД никак → в «хвосте» их не отличить от off-platform-плательщиков. (Бэкфилл exempt из FEAT-015 на проде отсутствует — либо откатан, либо не сохранился.)

## 3. paid_until (любой провайдер) + access_status

- **С `paid_until` вообще:** 19 (все — prodamus). Future 16, past 3. Других провайдеров нет (YooKassa/ручных дат — 0).
- **access_status:** active 53, paused_expired 3, paused_manual 3.
- Итого: из 53 активных доступов **только 16 подтверждены оплатой на платформе**. Остальные **37 активных — без платёжного следа** (из них 3 admin, 14 applicant без оплаты, и 20 платящих ролей без записи).

## 4. Сходится ли 19 + N ≈ 38?

**Не сходится «в лоб» — и это ожидаемо.** «19» — это не размер чат-когорты, а срез «кто прошёл через Prodamus». Разбор чат-когорты (leader+mentor как ядро Лиги):

| Роль | Всего | Active | Есть prodamus-active | exempt | paid_future |
|---|---|---|---|---|---|
| leader | 19 | 18 | 8 | 0 | 8 |
| mentor | 7 | 7 | 2 | 0 | 2 |

→ из 26 leader+mentor только **10** имеют prodamus. **16 leader/mentor активны без Prodamus.**

**Хвост (active, роль leader/mentor/intern, нет prodamus-active, не exempt) = 24 профиля:**
- intern — 9, leader — 10, mentor — 5.
- У всех `paid_until=NULL`, `last_prodamus_event=NULL`; 23 из 24 с привязанным TG (1 mentor без TG — Анна Минаева).

**Реконструкция под ~38 (гипотеза чат = активные ведущие/стажёры/менторы):**
active leader (18) + mentor (7) + intern (11) = **36** активных платящих-ролей. + пара admin/команды ≈ **38**. Т.е. чат ≈ «активные leader+mentor+intern + команда», а **не** «19 prodamus». Prodamus покрывает лишь ~10 из ядра + часть interns/applicants.

**Разрыв 38 vs 19 объясняется так:**
- 16 платят через Prodamus (валидно),
- ~24 платят **мимо платформы** (TargetHunter) / перенесены вручную / grandfathered,
- 3 admin — команда (бесплатно),
- 3 истёкших prodamus всё ещё могут висеть в чате.

**Остаётся реальный «неучтённый хвост» = 24 активных участника без billing-следа.** Их нельзя ни удалить, ни поставить на паузу по текущим данным — БД про их оплату ничего не знает.

## Следствия для миграции (важно)

1. **НЕ чистить/паузить по prodamus/paid_until** — выкосит 24 легитимных off-platform-плательщиков.
2. **Нужен ростер TG-чата со стороны бота** (BotHunter `ligaskrebeyko` / TargetHunter) → кросс-матч по `profiles.telegram` (есть `normalizeTelegramUsername`, одна функция на обе стороны). Только пересечение «в чате И платит» даёт безопасный список.
3. **Бартер/льготы не оцифрованы (exempt=0).** До миграции Ольге нужно пройтись по «хвосту» и пометить бартерных `auto_pause_exempt=true`, иначе при включении hard-block они попадут под паузу.
4. **Целевая модель это чинит:** платформо-инициированный checkout кладёт user_id в заказ → каждый будущий платёж однозначно привязан. Но **исторический хвост (24) миграция сама не заполнит** — их нужно либо (а) попросить оплатить через новую систему, либо (б) выставить `paid_until` вручную/бартер-флагом на переходный период.
