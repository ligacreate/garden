# Vite contenthash collision + Cache-Control: immutable = браузер залип на старом bundle

**Дата:** 2026-05-10.
**Тикет:** FEAT-016 (баг в процессе debug-сессии 2026-05-09).
**Связанный фикс:** commit `4fd8a4d` (тег console-логов с `report` →
`report v2` чтобы насильно сменить contenthash).

## Симптом

В FEAT-016 (admin per-student MD-отчёт + bulk ZIP) делал серию
точечных фиксов с детальной диагностикой. Через 2-3 итерации заметил:
**console-логи новой версии не появлялись в DevTools на проде**, при
этом git push прошёл, GitHub Actions deploy зелёный, FTP-deploy без
ошибок, на сервере новый `dist/` лежит.

Со стороны Ольги выглядело так:
- Hard reload (Cmd+Shift+R) не помогал.
- Console пуста, как будто новый код вообще не выкатился.
- Network tab показывал — `index-DvDJAQqT.js` стабильно тянется из
  disk cache.

Час дебага: пытался искать race condition, тупик в state, забытый
console.log в условном блоке. На самом деле **браузер просто не
качал новый bundle** — потому что URL bundle'а не изменился между
двумя deploy'ями.

## Корневая причина

Vite по умолчанию называет production-bundle через
`assets/index-[contenthash].js`, где contenthash считается от
**содержимого** chunk'а после минификации. Если diff между билдами
тривиальный (например, изменился порядок импортов в файле, добавлено
поле в стейт, но финальный минифицированный chunk имеет тот же AST) —
**contenthash может совпасть**. У нас два билда подряд получили
буквально один и тот же `index-DvDJAQqT.js`.

Дальше — конфигурация nginx hightek.ru (после INFRA-004):

```nginx
# /assets/*
add_header Cache-Control "public, immutable, max-age=31536000";
```

`immutable` означает «контент по этому URL **никогда не меняется**,
не делай revalidation request даже при F5 / Cmd+Shift+R». Это
правильный contract — bundle с контент-хешем в имени действительно
immutable. Но если новый deploy случайно отдал тот же имя —
браузер пользователя сидит со старой версией **до самого
истечения max-age (год)**.

То есть мы получили классическую коллизию:
- nginx правильно говорит «не перепроверяй»;
- Vite случайно дал старое имя;
- Browser честно не перепроверяет.

Никакой rebuild на сервере / clear FTP / `cache-busting` query string
не помог бы — URL остаётся тем же.

## Почему так получилось

1. **Слепое доверие к Cache-Control: immutable.** Контракт
   immutable требует, чтобы новый контент **гарантированно** имел
   новое имя. Vite контент-хеш в **большинстве** случаев это
   гарантирует, но не **всегда**. Узким местом стали тривиальные
   правки, не меняющие финальный output.
2. **Минификация может скушать diff.** Свернутые имена локальных
   переменных, удалённые мёртвые ветки, dead code elimination — два
   разных source-уровня правки могут дать идентичный chunk. С тех пор
   как мы делаем дебаг с минимальными правками («уточнил
   условие → одна строка», «переименовал поле → 3 use sites»), это
   происходит проще, чем кажется.
3. **Не было «канарейки» вида `BUILD_ID`/git SHA.** До MON-001 у
   нас не было способа быстро увидеть на странице — какая версия
   реально в браузере. Я смотрел на git log + Actions log и думал
   «новый код выкачен», а реально в браузере был старый.
4. **Hard reload (Cmd+Shift+R) **не пробивает immutable**.** Это
   контр-интуитивно (привычка: «Cmd+Shift+R обходит cache»), но
   `immutable` сильнее обычного `no-cache`. Только `Disable cache` в
   DevTools или явная очистка browser cache работают.

## Как починили

Точечный фикс — изменить тег console-логов в
[`views/AdminPvlProgress.jsx`](../../views/AdminPvlProgress.jsx) с
`'[FEAT-016 report]'` на `'[FEAT-016 report v2]'`. Это
гарантированно добавляет `' v2'` в минифицированную строку chunk'а →
другой contenthash → другое имя файла → браузер качает новое.

Реальная правка размером 4 байта, но цель не в правке — цель в
смене contenthash. Это **правильный workaround**, потому что:
- не требует менять nginx-конфиг (immutable — корректный режим);
- не требует менять Vite-конфиг;
- закрывает проблему за минуту.

После починки — добавили `BUILD_ID` через `vite define` (MON-001):
теперь в payload каждой ошибки есть `bundleId = process.env.GITHUB_SHA`,
и в смоках можно сравнивать «что в браузере» vs «что в репо» одной
строкой.

## Что проверить в будущем

### Паттерн для ловли похожих багов

**Сигналы, что вы попали в эту ловушку:**

- Свежий deploy зелёный, но новые console-логи не видны в DevTools
  пользователя.
- Network tab показывает bundle URL **точно такой же**, как в
  прошлом deploy'е (имя файла идентичное).
- Hard reload (Cmd+Shift+R) ничего не меняет.
- DevTools → Network → checkbox **Disable cache** включён → новый
  код наконец появился.

**Если хотя бы 2 из 4 — это immutable cache trap, не ваш код.**

### Профилактика

1. **Добавьте видимый build-id на страницу.** У нас теперь есть
   `__BUILD_ID__` через `vite define` (см. `vite.config.js`). На
   будущее: можно вывести его в footer / DevTools-метку.
   `console.log('build:', __BUILD_ID__)` в `main.jsx` — полминуты
   работы, экономит часы дебага.
2. **При тривиальных дебаг-правках сразу меняйте видимую строку.**
   Если правка — «уточнил условие в одной строке» и боитесь
   collision — поменяйте сразу что-то заметное в минифицированном
   chunk'е (имя console-тега, текст error message, добавьте
   `console.log('debug-pass-N')`). Это форсит новый contenthash.
3. **Smoke check на проде смотрит bundle URL.** Наш post-deploy
   smoke (`chore(ci): post-deploy smoke check`) проверяет, что
   bundle доступен по URL — но **не проверяет**, что URL изменился.
   Можно добавить: вытащить bundle URL из `dist/index.html` **до**
   FTP-deploy, сравнить с тем, что отдаёт прод сейчас, и зафейлить
   workflow если они одинаковые **и** мы реально меняли код. Это
   спорный шаг — false positives при «чисто docs»-deploy.
4. **Для критичных дебаг-сессий — `Disable cache` в DevTools.**
   Очевидно, но забывается. Когда смоук «не сходится» — первая
   проверка должна быть на свежесть bundle'а, а не на бизнес-логику.

### Тест на регресс

Тест-сценарий: два подряд коммита, второй меняет только пробелы в
файле, вне chunk-границ. Build → сравнить имена файлов в
`dist/assets/`. Если совпали — мы воспроизвели collision.

Не пишу автотест — слишком хрупкий контекст (зависит от поведения
Vite minifier и от того, что именно поменяли). Лучшая защита —
build-id в payload (готово в MON-001) + сигналы выше.

### Сигналы

- «Я уверен, что выкатил новый код, но Ольга/тестер не видит
  изменений» = **первым делом** Network → bundle URL → сравни с
  предыдущим. До любых других гипотез.
- Cache-Control: immutable + contenthash в URL — это правильный
  паттерн для bundle'ов **с гарантированно уникальным
  contenthash'ем**. Если используете immutable, гарантия на вашей
  совести (или на совести Vite, что в 99% случаев работает).
- При INFRA-004-стиле деплоях через FTP без CDN-purge — браузерный
  кэш у клиентов будет «хуже» серверного: nginx у hightek.ru
  обновится сразу, а у пользователя bundle сидит локально на год.
