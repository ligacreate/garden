Поиск по чек-листу задачи:

- хвостовой пробел / `&nbsp;` — **нет** (в БД чистый `<br>`, без пробела/nbsp-байта);

**Контраст с соседями (важно для диагноза).** У трёх других заданий Модуля 2 `<br>` стоит **после** `</pre>` (`…</pre><br>`) либо это нормальный HTML с `<p>/<br>`:

Сквозной precise-скан по всем `pvl_content_items` (и `body_html`, и `homework_config.prompt`) на «реальный тег внутри `<pre>`» (`~ '<pre[^>]*>[^<]*<[a-zA-Z]'`) даёт **ровно одну** строку — Урок 8. **Параллельных багов того же типа нет** (конспекты чистые).

## 2. Вердикт по вилке (а / б)

> а) экранированный/двойной HTML в БД (`&lt;br&gt;`) → баг СОХРАНЕНИЯ;
> б) в БД чистый `<br>`, на экране как текст → баг РЕНДЕРА (двойное escape при показе).

**→ Вилка (б): баг РЕНДЕРА.** В БД лежит **чистый реальный `<br>`** (не `&lt;br&gt;`), а на экране он показывается буквальным текстом. Двойное экранирование происходит **в момент показа** — в `normalizeMaterialHtml()`, в ветке для `<pre>`-«verbatim».

(Есть и вклад слоя сохранения — лишний `<br>` вообще не должен был попасть внутрь `<pre>` — но по определению вилки, раз значение в БД чистое, это рендер. См. §4.)

## 3. Сквозной путь

### 3a. РЕНДЕР участницам — здесь рождается literal `<br>`

[views/pvlLibraryMaterialShared.jsx:149-167](../../views/pvlLibraryMaterialShared.jsx#L149-L167) — `normalizeMaterialHtml`, ветка `<pre>`:

```js
const preMatch = raw.match(/<pre[^>]*>([\s\S]*?)<\/pre>/i);
if (preMatch) {
    const unwrapped = String(preMatch[1] || '')
        .replaceAll('&lt;', '<').replaceAll('&gt;', '>').replaceAll('&amp;', '&')
        .trim();
    const escaped = escapeHtml(unwrapped).replaceAll('\n', '<br/>');   // ← здесь
    return `<div class="pvl-doc-verbatim">${escaped}</div>`;
}
```

Трассировка для Урока 8 `prompt = "<pre>…на платформе.<br></pre>"`:
1. `preMatch[1]` = внутренность pre, **включая реальный `<br>`** (декод сущностей его не трогает — это уже `<br>`, а не `&lt;br&gt;`); `.trim()` его не срезает (`<br>` — не whitespace).
2. `escapeHtml(unwrapped)` превращает реальный `<br>` → `&lt;br&gt;`.
3. Возврат: `<div class="pvl-doc-verbatim">…на платформе.&lt;br&gt;</div>`.
4. Через `dangerouslySetInnerHTML` браузер показывает `&lt;br&gt;` как **буквальный текст `<br>`** → симптом.

Контракт ветки `<pre>` дырявый: она считает, что внутри `<pre>` лежит **простой текст** (для verbatim-документов: декодировать сущности, заэкранировать, `\n`→`<br/>`), и поэтому намеренно экранирует всё содержимое. Любой **реальный** inline-тег, попавший внутрь `<pre>`, выводится как текст. У соседей (Уроки 6,7) `<br>` стоит **после** `</pre>` и `normalizeMaterialHtml` возвращает только `preMatch[1]` → их `<br>` просто отбрасывается, поэтому у них чисто.

**Место показа задания:** [views/pvlLibraryMaterialShared.jsx:622](../../views/pvlLibraryMaterialShared.jsx#L622) (`HomeworkInlineForm`). Тот же `normalizeMaterialHtml` зовётся для конспекта ([:449](../../views/pvlLibraryMaterialShared.jsx#L449),[:460](../../views/pvlLibraryMaterialShared.jsx#L460),[:484](../../views/pvlLibraryMaterialShared.jsx#L484)) и в карточке ментора [views/PvlTaskDetailView.jsx:473](../../views/PvlTaskDetailView.jsx#L473) — фикс в одной функции чинит все точки показа.

### 3b. СОХРАНЕНИЕ — как `<br>` оказался внутри `<pre>`

- Задание правится через `RichEditor` (contentEditable): [views/PvlPrototypeApp.jsx:4794-4798](../../views/PvlPrototypeApp.jsx#L4794-L4798) (`key="lesson-hw-prompt"`, `value={hw.prompt}`).
- `RichEditor.sanitizeIncomingHtml` держит `PRE` и `BR` в whitelist ([components/RichEditor.jsx:43-49](../../components/RichEditor.jsx#L43-L49)) — `<pre>` грузится в contentEditable как есть.
- При ручном удалении в конце `<pre>`-блока браузер вставляет **bogus `<br>`** (служебный перенос, который нельзя «добэкспейсить») — это и есть «**неудаляемый пробел в конце**».
- `onInput`→`pushToParent` ([:181](../../components/RichEditor.jsx#L181)), `onBlur`→`flushSanitized` ([:189](../../components/RichEditor.jsx#L189)) сериализуют `innerHTML` как есть — лишний `<br>` сохраняется внутри `<pre>`.
- На сабмите `prompt` ещё раз проходит `sanitizeHomeworkAnswerHtml` ([views/PvlPrototypeApp.jsx:4674](../../views/PvlPrototypeApp.jsx#L4674)), у которого в `ALLOWED_TAGS` есть и `pre`, и `br` ([utils/pvlHomeworkAnswerRichText.js:18-21](../../utils/pvlHomeworkAnswerRichText.js#L18-L21)) → `<br>` внутри `<pre>` остаётся.
- Итог в БД: `<pre>…<br></pre>`.

> Происхождение самого `<pre>`-обёртки: не из `data/pvl/seed.js` (там нет) и не из md-импорта (`utils/pvlMarkdownImport.js` гонит через `marked` → `<p>`, а не `<pre>`). Похоже на разовый verbatim-импорт курса (пошаговый текст обёрнут в `<pre>`, чтобы `marked` не ломал нумерацию шагов) — ровно под это в `normalizeMaterialHtml` и сделана `<pre>`-ветка. Это контекст, не корень бага.

**Два проявления одного артефакта:**
- в редакторе (учительская): bogus `<br>` внутри `<pre>` = «неудаляемый пробел в конце»;
- у участницы/ментора: тот же `<br>` экранируется при показе = «буквальный `<br>`».

## 4. Вердикт: где дублируется и точечная починка

**Где именно.** Чистый реальный `<br>` дублируется/экранируется на показе — в `normalizeMaterialHtml`, ветка `<pre>` ([views/pvlLibraryMaterialShared.jsx:152-161](../../views/pvlLibraryMaterialShared.jsx#L152-L161)): `escapeHtml(unwrapped)` экранирует реальный inline-тег, который ветка не нормализовала. Это owner-слой трансформации «хранимый HTML → показанный HTML», и чинить надо здесь (не компенсацией в дочернем слое).

**Предлагаемый фикс (основной, рендер — вилка б).** В `<pre>`-ветке нормализовать реальные `<br>` в перенос строки ДО экранирования, чтобы `\n`→`<br/>` затем восстановил корректные переносы, а хвостовой срезался `.trim()`:

```diff
     const unwrapped = String(preMatch[1] || '')
         .replaceAll('&lt;', '<')
         .replaceAll('&gt;', '>')
         .replaceAll('&amp;', '&')
+        .replace(/<br\s*\/?>/gi, '\n')   // реальный <br> внутри verbatim-pre (артефакт contentEditable) → перенос строки
         .trim();
     const escaped = escapeHtml(unwrapped).replaceAll('\n', '<br/>');
```

Почему так, без слепого патча:
- Сохраняет назначение `<pre>`-ветки (verbatim-документы рендерятся как и раньше), но закрывает дырку в контракте «внутри pre — простой текст».
- Чинит **уже сохранённые данные** (единственную строку Урока 8) **без миграции** и сразу во всех точках показа (библиотека/трекер/ментор).
- Хвостовой `<br>` у Урока 8 после `<br>`→`\n`+`.trim()` исчезает (концовка `…на платформе.` чистая); `<br>` в середине verbatim-дока стал бы корректным переносом, а не literal-текстом.
- Порядок (декод сущностей → `<br>`→`\n` → trim → escape) ловит и реальный `<br>`, и гипотетический `&lt;br&gt;`.
- Риск: если verbatim-док НАМЕРЕННО показывает текст-тег `<br>` (туториал по HTML) — он станет переносом. Для прозы курса нереалистично; в БД таких кейсов нет.

**Защитный фикс (вторичный, сохранение — предотвратить рецидив).** Срезать хвостовой bogus `<br>` — последний дочерний узел блока (особенно `<pre>`) — в `RichEditor.normalizeEditorHtml`/`sanitizeIncomingHtml`. Убирает «неудаляемый пробел» в самом редакторе. Шире по влиянию (трогает редактор для всех типов контента; contentEditable иногда требует хвостовой `<br>` для каретки) → **отдельным, аккуратным шагом**, не блокирует основной фикс.

**Точечная чистка данных (опционально).** Разовый `UPDATE` строки `3d090f4c…` (убрать `<br>` перед `</pre>`) — после основного рендер-фикса становится косметикой и **не обязателен**. Если делать — с защитой от двойного прогона (`WHERE … LIKE '%<br></pre>%'`).

**Рекомендация:** основной фикс в `normalizeMaterialHtml` (вилка б, owner-слой, data-independent). Защитный фикс редактора — следующим тикетом.
