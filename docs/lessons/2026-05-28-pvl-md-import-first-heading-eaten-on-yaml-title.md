# Парсер `.md` съел первый `##` из тела, потому что не учёл, что title пришёл из YAML

**Дата инцидента:** 2026-05-28 (обнаружение + fix + deploy).
**Связанная миграция:** ни одной (frontend-only + одноразовый UPDATE одной записи).
**Связанные коммиты:** `d415311` ([utils/pvlMarkdownImport.js:110](utils/pvlMarkdownImport.js#L110)).
**Связанные сессии:** [_141 recon](../_session/2026-05-28_141_codeexec_recon_cms_headers_lost.md), [_142 apply diff](../_session/2026-05-28_142_codeexec_fix_cms_first_heading_lost_diff.md).

## Симптом

В материале курса ПВЛ «Драматургия встречи — динамика, ритм, разнообразие» все четыре последующих `<h2>` (Принцип второй / третий / Итак… / Микро-задание) рендерились корректно, но **самый первый — «Принцип первый. Динамика» — отсутствовал в БД целиком**. Тело начиналось сразу с `<p>Хороший сценарий — это…</p>`.

Жалоба админа: «Почему-то этот заголовок не добавился. Хотя я добавляла файл и вручную не получается.» — то есть и через `.md` upload, и через RichEditor ничего не выходило (вторая часть — отдельный open вопрос про RichEditor).

## Корневая причина

[`utils/pvlMarkdownImport.js:81-134`](../../utils/pvlMarkdownImport.js#L81-L134), функция `parsePvlImportedMarkdownDoc`:

```js
let title = String(yamlTitle || '').trim();
let headingIndex = -1;
for (let i = 0; i < lines.length; i++) {
    const hm = lines[i].trim().match(/^#{1,6}\s+(.+)$/);
    if (hm) {
        if (!title) title = hm[1].trim();   // ← title заполняется только если пустой
        headingIndex = i;                    // ← но index запоминается ВСЕГДА
        break;
    }
}
...
if (headingIndex >= 0) {                     // ← BUG: вырезает heading из тела
    nl.splice(headingIndex, 1);              //   даже когда title пришёл из YAML
    mdBody = nl.join('\n');
}
```

**Цепочка отказа для файла с YAML + первым `##`:**
1. `.md` приходит с `---\ntitle: Драматургия…\n---` + `## Принцип первый. Динамика` как первый ATX-heading в теле.
2. `peelYamlFrontMatter` → `yamlTitle = "Драматургия…"`, `title` уже занят.
3. Цикл находит `## Принцип первый. Динамика` → `title` НЕ перезаписан (защита `!title` сработала), но `headingIndex = i` записан.
4. Безусловный `if (headingIndex >= 0) splice` — **вырезает `## Принцип первый.` из тела**, хотя этот heading не использовался для title.
5. `marked.parse(remaining)` — отдаёт HTML без первого `<h2>`. Остальные `## Принцип второй.`, `## Принцип третий.` сохраняются.
6. HTML сохраняется в `body_html` → в БД отсутствует первый принцип.

## Почему так получилось

**Antipattern:** «обнаружил → пометил → действие» без сверки **зачем** пометил. Цикл записывает `headingIndex` как универсальный сигнал «первый ATX-heading найден», но действие `splice` валидно **только** когда этот heading реально стал title. Связь «пометка → действие» некритически разорвалась как только появился второй источник title (YAML frontmatter).

**Латентность:** `peelYamlFrontMatter` добавлялся отдельно от цикла поиска heading. Без YAML frontmatter обе ветки работают (heading → title → вырезается из тела). С YAML — `if (!title) title = hm[1]` корректно защищает title, но `splice` ниже об этом не знает. Каждая половина в одиночку выглядит правильной. Регрессия не возникает в типичных тест-кейсах: `.md` без frontmatter работает нормально.

**Маскирование:** в кодовой базе нет seed-материалов с YAML, и большинство ручных импортов админа тоже без него. Баг проявился только когда Ольга экспортнула материал из Notion с `title:` в frontmatter.

## Как починили

`d415311` — одна строка:

```diff
-    if (headingIndex >= 0) {
+    if (headingIndex >= 0 && !yamlTitle) {
         nl.splice(headingIndex, 1);
```

`else if (yamlTitle && lines.length)` ветка ниже уже умеет аккуратно убрать первую строку, **но только если она совпадает с title** (защита от дубля). Все остальные сценарии (YAML + первый `##` который ≠ title) теперь корректно сохраняют heading в теле.

Для уже сохранённой «Драматургии» в БД — одноразовый `UPDATE … SET body_html = REPLACE(...)` с защитой `body_html NOT LIKE '%<h2>Принцип первый. Динамика</h2>%'` от двойного прогона.

## Что проверить в будущем

### Pattern: «обнаружил → пометил → безусловное действие»

Если цикл записывает индекс/флаг для «универсального» использования, а действие на этом индексе **валидно только при одном из путей в цикле** — это будущий баг. Особенно когда в условии действия можно условно нарушить (новый источник данных, новая ветка). Эвристика:

```js
for (...) {
    if (cond) {
        if (!acquired) acquired = take();   // ← брать только если ещё не брали
        index = i;                           // ← но index пишется ВСЕГДА
        break;
    }
}
...
if (index >= 0) { sideEffect(index); }       // ← валидно ли «всегда»?
```

Спроси: «при каких ветках цикла `sideEffect` НЕ должен исполняться?» Если хотя бы одна — добавь то же условие что у `acquired`.

### Pattern: парсер с двумя источниками одного слота

Title для импортированного документа имеет минимум три источника: YAML frontmatter, первый ATX heading, имя файла. У каждого источника свой «связанный side-effect» (вырезание из тела, нормализация title, fallback). Когда добавляется новый источник — связи между источниками и side-effect'ами надо ре-валидировать **парно**. Не хватает test-cases с пересечением источников.

**Действие:** при следующем касании парсера — добавить unit-тест на матрицу комбинаций (YAML +/- ATX +/- file-name) с проверкой что body содержит ожидаемые headings (или не содержит).

### Pattern: визуальная асимметрия как сигнал

Симптом «второй/третий/четвёртый заголовок есть, первый отсутствует» — это **прямой указатель на off-by-one / first-iteration bug** в каком-то цикле. Если в дате-driven UI видишь регулярный паттерн с дыркой ровно в первом элементе — ищи не «почему первый не такой», а «что обрабатывается специально для первого». В этом случае: `break` после первого find + безусловный `splice` ровно на этом первом.

### Pattern: hard reload решает loaded-once-cached UI state

После любого UPDATE на проде у пользователя в открытой админке может оставаться **закешированный в React-state** объект материала, прочитанный ДО UPDATE. F5 / hard reload фиксирует. Не баг кода — но если пользователь жалуется «не работает» сразу после migration — сначала спроси про reload.
