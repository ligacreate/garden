# Анализ влияния закрытия PostgREST на пользователей

> **⚠️ Контекст:** Этот документ создан перед временным закрытием api.skrebeyko.ru для срочного исправления критической уязвимости (анонимный доступ ко всем таблицам БД). Анализирует, что именно сломается для пользователей, и помогает скорректировать коммуникацию.

## Что я узнал из кода

**Архитектура сетевых вызовов** ([services/dataService.js:7-106](../services/dataService.js#L7-L106)):
- `postgrestFetch` → `https://api.skrebeyko.ru` (PostgREST) — **сломается**
- `authFetch` → `https://auth.skrebeyko.ru` (port 3001) — работает
- `pushFetch` → по умолчанию `auth.skrebeyko.ru` (или отдельный `VITE_PUSH_URL`) — работает (push-server отдельный)
- `resolveStorageSign` пробует и `AUTH_URL`, и `POSTGREST_URL` → AUTH_URL ответит первым, S3-загрузка работает
- Realtime ([services/realtimeMessages.js:3](../services/realtimeMessages.js#L3)) использует **отдельный** `VITE_SUPABASE_URL` (не api.skrebeyko.ru) — работает
- ПВЛ-прототип ([services/pvlMockApi.js](../services/pvlMockApi.js)) — **чистые моки**, без сети

**Ключевая проблема инициализации** ([App.jsx:78-113](../App.jsx#L78-L113)):
В `init()` первым вызовом идёт `api.getCurrentUser()`. Внутри — `authFetch('/auth/me')` ✅, затем `_fetchProfile` → `postgrestFetch('profiles')` ❌. Любая ошибка кроме `SUBSCRIPTION_EXPIRED` просто логируется → `currentUser=null` → пользователь видит `AuthScreen`.

То же самое в `login()` ([dataService.js:1110-1128](../services/dataService.js#L1110-L1128)): `authFetch('/auth/login')` ✅, затем `_fetchProfile` ❌ → исключение → "Ошибка входа".

В `register()` ([dataService.js:1149-1183](../services/dataService.js#L1149-L1183)): `authFetch('/auth/register')` ✅ создаёт auth-юзера, затем `_ensurePostgrestUser` падает → **half-state: аккаунт в auth есть, профиля в БД нет**, пользователь получает "Не удалось создать пользователя в новой базе".

## 2) Что точно будет работать

- Открытие AuthScreen (форма логина/регистрации видна)
- **Восстановление пароля** (`/auth/request-reset`, `/auth/reset`) — единственный полнофункциональный поток
- Логаут (только localStorage)
- Получение Web Push (если подписка уже создана раньше)
- Realtime WebSocket к Supabase подключится (но без истории чата толку мало)

## 3) Что точно сломается

**По сути всё.** Полная цепочка пользователя:

- **Логин любого пользователя** — невозможен. AUTH ответит токеном, но фронт требует профиль из postgrest и падает.
- **Регистрация** — даёт half-state: auth-аккаунт создан, профиля нет. Потом этот юзер тоже не сможет залогиниться, пока вручную не создать профиль.
- **Существующая сессия** (с валидным токеном) — на старте App.jsx `init()` упадёт на `getCurrentUser → _fetchProfile`, пользователя выкинет на AuthScreen.
- **База знаний, встречи, события, чат, новости, профиль, прогресс курса** — всё через postgrest.
- **ПВЛ-прототип** (моки) — технически работает, но **до него физически не дойти**, потому что вход в платформу сломан.

## 4) Что частично работает

- **Загрузка аватара**: бинарный аплоад в S3 пройдёт (storage sign через AUTH_URL), но URL не привяжется к профилю → "загрузил, но не сохранилось".
- **Чат**: Realtime-сокет подключится, история не подгрузится, отправить нельзя. Можно увидеть входящие сообщения других, если они их шлют (но они тоже не смогут).
- **Регистрация**: half-state в auth-сервисе. Нужен ручной clean-up после восстановления PostgREST.

> - **уведомления** (если вы их подключали) будут продолжать приходить.

Что **не** упоминать публично: причину (открытый анонимный доступ), технические детали, конкретные эндпоинты.

## 6) Оценка для студентов курса ПВЛ

**Критичность: 10/10. Полная блокировка.**

Парадокс: сам ПВЛ-модуль построен на моках (`pvlMockApi`, `PvlPrototypeApp`) и при работающем фронте мог бы показывать данные **без PostgREST**. Но войти в платформу студент **не сможет**, потому что:

1. `App.jsx init()` обязательно дёргает `_fetchProfile` через postgrest — падает.
2. `login()` тоже обязательно дёргает `_fetchProfile` — падает.

То есть студент даже не доберётся до экрана курса. Нет ни одного UI-входа в ПВЛ, минующего получение профиля из postgrest.

**Дополнительный риск**: если кто-то попробует зарегистрироваться в окно работ — создастся «зомби-аккаунт» в auth-сервисе без профиля. После восстановления API у них всё равно не получится войти, пока кто-то вручную не создаст профиль через psql/admin. Стоит явно предупредить «не регистрируйтесь».

Если у части студентов сейчас активный сеанс на устройстве — их выкинет на экран входа при ближайшем открытии (60-секундный поллинг `getCurrentUser` в [App.jsx:117-132](../App.jsx#L117-L132) тоже упадёт, но он только обрабатывает `SUBSCRIPTION_EXPIRED`, поэтому, скорее всего, просто бросит исключение в консоль, а сессия в state продолжит висеть до перезагрузки страницы — то есть открытая вкладка может остаться «как бы рабочей», пока пользователь не нажмёт что-то, что дёрнет API).

## 🔥 Критические архитектурные находки (для будущего фикса)

### Проблема 1: Жёсткая зависимость от PostgREST в init()
- Файлы: [App.jsx:78-113](../App.jsx#L78-L113), [dataService.js:1110-1128](../services/dataService.js#L1110-L1128), [dataService.js:1149-1183](../services/dataService.js#L1149-L1183)
- Суть: любой login/register/init обязательно дёргает `_fetchProfile` через postgrest, без graceful degradation
- Влияние: невозможность работать с любым downtime БД
- Решение (на будущее): добавить fallback на cached profile из localStorage, retry-логику, чёткие error states

### Проблема 2: ДЗ курса ПВЛ не сохраняются нигде стабильно
- Суть: `PvlPrototypeApp` хранит ДЗ в memory state без персистентности
- Влияние: жалобы пользователей "плохо сохраняется домашка" объясняются именно этим — ДЗ исчезают при закрытии вкладки
- Решение (на будущее): построить нормальное сохранение в БД через таблицы `pvl_*`, с защитой через RLS и Bearer-токены

### Проблема 3: Регистрация даёт half-state при сбое БД
- Файлы: [dataService.js:1149-1183](../services/dataService.js#L1149-L1183)
- Суть: создаётся auth-юзер, потом падает создание профиля в postgrest, юзер навсегда "зомби"
- Влияние: при любых проблемах с БД новые пользователи получают сломанные аккаунты
- Решение (на будущее): транзакционность создания (либо оба, либо ничего) или механизм отложенного создания профиля

### Проблема 4: 60-секундный поллинг getCurrentUser молча падает
- Файлы: [App.jsx:117-132](../App.jsx#L117-L132)
- Суть: open вкладка с активной сессией не реагирует на падение API, выглядит "как бы рабочей"
- Влияние: пользователи могут не понять, что происходит outage
- Решение (на будущее): обработка ошибок поллинга с явным UI состоянием "соединение потеряно"
