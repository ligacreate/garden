## 1. Модель в одном абзаце
**Платформенный аккаунт — единственная парадная дверь.** Любой путь к Лиге идёт через кабинет:
`регистрация → одобрение админом (роль) → человек В кабинете → Лига-поверхности заперты замком до оплаты, а курс/обучение работает по роли и замком НЕ трогается → оплата → Лига разблокируется`.
Off-platform-без-аккаунта как популяция **закрывается** (все проходят регистрацию); легаси-плательщики без профиля — разово мигрируются.

## 2. Инвариант «кабинет-первый»
- Нет Лига-доступа без платформенного профиля. Точка входа в Лигу — кабинет, не Prodamus/TargetHunter.
- Оплата ≠ платформенный доступ. Это **два независимых слоя** (см. §4). Курс живёт на первом слое (роль), Лига — на втором (оплата).

## 4. Целевая модель — ДВА независимых слоя доступа

| Слой | Ключ | Что открывает | Кто ставит |
|---|---|---|---|
| **A. Платформенный доступ** | `access_status='active'` (RLS `has_platform_access`) | вход в кабинет, курс/обучение **по роли** | одобрение админом (§3.1) |
| **B. Лига-подписка** | `subActive = paid_until >= now` | Лига-поверхности: TG-кнопки, Лига-сообщество, «замо́к» снят | оплата (webhook/mark-paid) |

**Правило:** курс живёт на слое A (роль). Лига живёт на слое B (оплата). **Неоплата Лиги НЕ должна ронять слой A** (иначе отнимаем курс — нарушение «курс-не-трогать» + ГАРДРЕЙЛА плана).

## 5. ⚠️ ГЛАВНОЕ НАПРЯЖЕНИЕ (требует твоего решения)

Сейчас Лига-неоплата → авто-пауза `runNightlyExpiryReconcile` → `access_status='paused_expired'` → RLS **запирает ВЕСЬ кабинет** → человек теряет и курс, и возможность оплатить внутри. Это **прямо конфликтует** с «кабинет-первый» (замок только на Лиге, курс не трогаем).

**Три варианта развязки (рекомендую В1):**

- **В1 (рекомендую) — расцепить слои: неоплата Лиги НЕ меняет `access_status`.**
  `access_status='active'` остаётся у одобренного всегда (пока админ не выключил вручную). Лига-замок = чисто `subActive`-гейт на Лига-поверхностях. Автопаузу по `paid_until` (`paused_expired`) **выводим из строя** для платящих ролей ИЛИ переносим её эффект с «весь кабинет» на «только Лига-флаг».
  *Плюс:* курс и вход в кабинет неприкосновенны, оплата доступна изнутри. *Минус:* `paused_expired` как состояние обесценивается; надо переосмыслить reconcile.

- **В2 — оставить RLS-паузу, но вынести курс и billing из-под `has_platform_access`.**
  Курс-таблицы и тарифы получают отдельную, роле-based RLS без `access_status`-гейта. *Минус:* хирургия по 39 RLS-политикам, высокий риск, «курс-не-трогать» превращается в точечные исключения — хрупко.

- **В3 — ничего не менять в access_status, замок Лиги только на фронте.**
  *Минус:* фронт-замок обходится; не источник правды. Не годится для платного контента.

**Решение по В1/В2/В3 — блокер стройки.** Дальше §6–§9 расписаны в предположении **В1**.

## 6. Поток пользователя (модель В1)

1. **Регистрация** → `pending_approval` (как сейчас, §3.1). Кабинет закрыт RLS.
2. **Одобрение админом** → `access_status='active'` + роль. Человек в кабинете; **курс по роли работает сразу**.
3. **Лига-замок** (новое): Лига-поверхности показывают «замо́к» + CTA «Оплатить», пока `!subActive`. Курс — без замка́.
4. **Оплата** (кабинет-checkout Prodamus, §3.6) → webhook (товаро-гейт уже на проде, 253) → `paid_until` → `subActive=true`.
5. **Разблок**: Лига-кнопки «Вступить в канал/чат» становятся активны; поллер авто-одобряет заявку (профиль есть → матч по uid/username).

## 7. Доставка join-ссылок (из 251, оба канала Лига-гейтед)

**Нюанс каталога (§3.6):** платформенный checkout строит товар ad-hoc → поле «Платный контент» к нему НЕ привязано → email-доставка от Prodamus для нашего checkout **не сработает**. Развязки (решить):
- (a) checkout ссылается на **каталожный** Лига-товар (id) вместо ad-hoc `products[0][name]`; ЛИБО
- (b) своя **SMTP-рассылка** join-ссылок после успешного `plan_payment`, гейт по `order.plan_code ∈ Лига` (товаро-безопасно, `plan_code` уже есть).
Belt-and-suspenders: standing-кнопки в кабинете (a/b — для email-подстраховки и off-platform).

## 8. Off-platform без профиля + миграция легаси
- **Закрытие как популяции:** «кабинет-первый» = регистрация обязательна → новых off-platform-без-профиля не появляется (recurring TH дорабатывает, cutover позже).
- **Замок доставки ≠ впуск:** плательщик без профиля кликнет standing-ссылку → заявка → поллер **не одобрит** (unknown). Нужен профиль ⇒ **миграция легаси**: свести существующих Лига-плательщиков без `profiles`-row (webhook 202-replayable хвост + recurring TH) к профилю — либо пригласить зарегистрироваться, либо админ заводит профиль и матчит платёж. Только после профиля поллер их сматчит.
- Сверка объёма хвоста — read-only запрос перед стройкой (см. §11).
