# FEAT-023 — Регистрация по одобрению админа

**От:** стратег (claude.ai)
**Кому:** VS Code Claude Code
**Тип:** recon + план, до кода.

---

## Проблема

Ольга обнаружила security hole: **любой человек с ссылкой на liga.skrebeyko.ru может зарегистрироваться, автоматически получить роль `applicant`, и тут же иметь доступ к курсу ПВЛ + платформе**. Закрытое сообщество получает «открытую дверь».

## Решения Ольги (2026-05-16)

1. **Pending screen для нового юзера:** «Администратор скоро предоставит вам доступ к платформе.» — без доступа к контенту.
2. **На одобрении:** админ выбирает роль из списка (applicant/intern/leader/mentor).
3. **На отклонении:** профиль удаляется полностью (используй существующий `admin_delete_user_full(uuid)` RPC из phase24).
4. **TG-уведомление:** в существующий бот `@garden_grants_monitor_bot` (тот же endpoint что MON-001).
5. **Содержание уведомления:** имя + email + город (если указан) + прямая ссылка на профиль в админке для быстрого одобрения.

## Что нужно от тебя — recon + план, БЕЗ кода

### 1. Recon текущего register-flow

- Где в `garden-auth/server.js` (или `views/Auth*.jsx`) находится endpoint регистрации? Какие поля принимает (email, password, name, city, role)?
- Какая роль присваивается по умолчанию сейчас? (Вероятно `applicant`, но подтверди.)
- Что записывается в `profiles` при регистрации? Какие таблицы ещё (users_auth, etc)?
- Есть ли проверка дубликата email сейчас?

### 2. Дизайн pending state

Два варианта реализации:

- **Вариант A: новая роль `pending`** в `profiles.role`. Добавить в CHECK constraint, дефолт для регистрации. Frontend проверяет `role === 'pending'` → показывает заглушку.
- **Вариант B: отдельный флаг `is_approved boolean default false`** в `profiles`. Дефолт false при регистрации, true при одобрении. Роль остаётся на момент регистрации (например `applicant`), но без `is_approved` доступа нет.

Подумай и предложи — какой чище. **Я бы взяла A** (одно поле `role`, проще для RLS-политик), но твоё мнение важно если есть подводные.

### 3. Backend изменения

- При регистрации: устанавливать `role='pending'` (или `is_approved=false`) вместо текущего `applicant`.
- При одобрении: endpoint типа `POST /api/admin/approve-registration` с параметрами `{user_id, new_role}`. Меняет роль на выбранную. Логирует в `pvl_audit_log` (или подобный).
- При отклонении: вызывать существующий `admin_delete_user_full(user_id)` RPC.
- **TG-уведомление при регистрации:** через `reportClientError` или новый отдельный endpoint? Скорее всего отдельный, например `POST /api/admin/notify-new-registration`. Шлёт в тот же endpoint что MON-001 (или напрямую в `@garden_grants_monitor_bot`).

### 4. Frontend изменения

- **Pending screen:** если у залогиненного юзера `role === 'pending'` (или `is_approved === false`) — показать минимальный экран с сообщением «Администратор скоро предоставит вам доступ к платформе.» Никаких меню/контента. Кнопка «Выйти».
- **Admin UI:** новая вкладка «Ожидают одобрения» в AdminPanel:
  - Список pending-юзеров: имя, email, город (если есть), дата регистрации.
  - На каждой строке кнопки «Одобрить» / «Отклонить».
  - При «Одобрить» — модалка с выбором роли (radio applicant/intern/leader/mentor) и подтверждением.
  - При «Отклонить» — confirm-dialog «Удалить регистрацию?» (предупреждение что профиль удалится навсегда).
  - Счётчик в шапке вкладки: «Ожидают одобрения (N)».

### 5. RLS implications

- Pending-юзер не должен видеть ничего из PVL/meetings/practices. Текущие RLS почти все требуют `role IN ('applicant','intern','leader','mentor','admin')` — проверь, что pending туда не входит, и если входит — закрой явно.
- Pending-юзер может писать в свой профиль (например, обновить имя/город) или нет? Если да — узкая полоска прав. Если нет — pending = только read own profile, никаких других прав.

### 6. Защита от race condition / re-register

- Что если человек дважды нажал «Зарегистрироваться»? Должен ли получить ошибку «email уже занят» или второй раз попадёт в pending?
- Что если pending-юзер удалён (Reject), а потом снова регистрируется тем же email — заходит снова в pending? Или блокируем?

Предложи разумную дефолтную логику.

С количеством сессий на каждую фазу.
