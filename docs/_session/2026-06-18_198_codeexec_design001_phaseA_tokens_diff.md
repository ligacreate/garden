## 1. TL;DR

Добавил в `index.css @theme` 8 цветовых токенов тёплой палитры + 3 кривые easing (значения 1:1 со спекой). Собрал, **побайтово сравнил скомпилированный CSS до/после** (чистый билд с обеих сторон).

**Результат не «diff = 0», как требует acceptance Фазы A.** Нашёл сюрприз: **два из трёх easing-токенов (`--ease-out`, `--ease-in-out`) — зарезервированные имена в Tailwind v4 и переопределяют встроенные кривые**, которыми пользуются уже существующие утилиты `.ease-out` (2×) и `.ease-in-out` (3×) в `AuthScreen`, `PvlStudentTrackerView`, `UserApp`. Их кривые в собранном CSS поменялись → это поведенческий diff на 5 существующих анимациях. Цветовые токены — наоборот, **идеально безопасны** (Tailwind их tree-shake'ает, в выводе их нет вообще).

**Поэтому не коммичу как есть. Нужно решение стратега по неймингу easing (вопрос внизу).**

## 2. Замер (чистый билд до → чистый билд после)

Метод: `npm run build` на исходном `index.css` → сохранил CSS; применил правку → `npm run build` → сохранил CSS; нормализовал (`}` → перенос строки) и `diff`. Сборка зелёная в обоих случаях.

Полный нормализованный diff = **6 строк**, и они делятся ровно так:

### ✅ Цветовые токены — ноль импакта (tree-shaken)
`--color-surface / -soft / -line / -ink / -ink-strong / -ink-soft / -ink-mute / -clay`
→ **0 вхождений** в скомпилированном CSS. Tailwind v4 эмитит `@theme`-переменные цвета только если их использует хоть одна сгенерированная утилита. Ни одной (`bg-surface` и т.п. нигде нет — это Фаза D). Визуальный и CSS-diff от них = **0**. Ровно как и задумано «определены, но не применены».

### ✅ `--ease-drawer` — безопасен
Имя `drawer` НЕ совпадает ни с одной встроенной кривой Tailwind → ничего не переопределяет. В выводе появляется как неиспользуемая `--ease-drawer:cubic-bezier(.32,.72,0,1)` в `:root`. Утилита `.ease-drawer` не эмитится (не используется). Импакт = 0.

### ❌ `--ease-out` и `--ease-in-out` — КОЛЛИЗИЯ со встроенными Tailwind v4
Tailwind v4 из коробки определяет `--ease-out`, `--ease-in-out`, `--ease-in` и генерит из них утилиты `ease-out` / `ease-in-out`. Переопределение значения в нашем `@theme` поменяло уже существующие утилиты:

```diff
- .ease-out{animation-timing-function:cubic-bezier(0,0,.2,1)}
+ .ease-out{animation-timing-function:cubic-bezier(.23,1,.32,1)}
- .ease-in-out{animation-timing-function:cubic-bezier(.4,0,.2,1)}
+ .ease-in-out{animation-timing-function:cubic-bezier(.77,0,.175,1)}
```

Эти классы реально используются:
- `views/AuthScreen.jsx`
- `views/PvlStudentTrackerView.jsx`
- `views/UserApp.jsx`
- (всего: `ease-out` 2×, `ease-in-out` 3×)

→ 5 существующих анимаций сменили кривую. **Это нарушает acceptance Фазы A «визуальный diff = 0 / не менять ни одного места использования».**

## 3. Что предлагаю (нужно 🟢 на вариант)

Цветовые токены — оставить ровно как в спеке (доказано: ноль импакта).
`--ease-drawer` — оставить как в спеке (безопасен).
Развилка только по двум коллизионным именам:

**Вариант A (рекомендую для Фазы A — сохраняет diff=0):** переименовать два коллизионных токена в не-зарезервированные имена, чтобы `.ease-out`/`.ease-in-out` остались на дефолтах Tailwind. Например:
```
--ease-soft:    cubic-bezier(0.23, 1, 0.32, 1);   /* было --ease-out  */
--ease-smooth:  cubic-bezier(0.77, 0, 0.175, 1);  /* было --ease-in-out */
--ease-drawer:  cubic-bezier(0.32, 0.72, 0, 1);
```
В Фазах C/D кнопки/поля будут получать `ease-soft` / `ease-drawer` — те же кривые, что в плане, просто под нейтральными именами. Diff Фазы A = 0 строго.

**Вариант B (НЕ для Фазы A):** оставить имена `--ease-out` / `--ease-in-out` и принять, что мы СОЗНАТЕЛЬНО глобально переопределяем дефолтную кривую Tailwind на тёплую во всём приложении. Это легитимное дизайн-решение, но это уже Фаза C (полировка ощущения), и оно ломает гарантию «diff=0» Фазы A. Если хотим — выносим в отдельный коммит Фазы C с осознанным «да, меняем 5 анимаций».

**Моя рекомендация:** Вариант A сейчас (Фаза A честно zero-diff), а вопрос «не сделать ли warm-кривую глобальным дефолтом» вынести в Фазу C, где он по смыслу и живёт.

## 5. Что НЕ затронуто

- Ни одного файла в `views/` или `components/` — никаких замен `bg-[#...]` → `bg-surface`. Это Фаза D.
- Существующие токены `--color-slate-*`, `--color-blue-*` и пр. — не тронуты.
- Существующие `.ease-out`/`.ease-in-out` утилиты — при Варианте A остаются на дефолтах Tailwind (это и есть смысл переименования).

## 6. Edge-case'ы

- **html2pdf:** новые цветовые токены — sRGB hex, как и slate/blue (тот же мотив, что комментарий «HEX вместо OKLCH»). Совместимо.
- **Tailwind namespace:** проверено билдом, что `ease-soft`/`ease-smooth`/`ease-drawer` не коллизируют (нет встроенных). `ease-out`/`ease-in-out`/`ease-in` — коллизируют (есть встроенные).
