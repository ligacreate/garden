# Сад ведущих — навигация по проекту

Этот документ — карта проекта. Читай его в начале каждой сессии, чтобы понимать, где что лежит и как с этим работать.

## Что это за проект

**Сад ведущих** — платформа-экосистема для ведущих письменных практик: инструменты, сообщество и личный рост в одном месте.

Помогает ведущим развиваться профессионально, находить коллег и вести практики на высоком уровне. Основная воронка: человек проходит обучение на ведущего → переходит на платформу по подписке.

Подробности — в [.business/INDEX.md](.business/INDEX.md).

## Ключевые принципы проекта

- **Сообщество важнее продукта** — живые связи между ведущими важнее алгоритмов и метрик
- **Личный рост ведущего** — жемчужина платформы, не просто библиотека контента
- **Первые в нише** — прямых конкурентов-платформ нет, мы задаём стандарт

## Стек

**Frontend:** React 19 + Vite 7 + Tailwind CSS 4 + JavaScript (ES Modules)

**Backend:** PostgreSQL + PostgREST API (`api.skrebeyko.ru`) + Auth Service (`auth.skrebeyko.ru`)

**Push-сервер:** Node.js + Express 5 + Web Push (VAPID), порт 8787

**Платежи:** Prodamus (webhook-интеграция)

**Деплой:** GitHub Actions → FTP на `liga.skrebeyko.ru`

**Инфра:** `VITE_POSTGREST_URL`, `VITE_AUTH_URL`, `VITE_PUSH_URL`, `VITE_WEB_PUSH_PUBLIC_KEY`

## Как работать с проектом

- Перед любой задачей — прочитай этот файл, затем нужные документы из `.business/` и `plans/`
- Если меняется бизнес-логика — обнови `.business/INDEX.md` и нужные файлы
- Если меняется технический план — обнови соответствующий план в `plans/`
- Если появляется новая важная папка или документ — обнови этот `CLAUDE.md`

## ВАЖНО: план для каждой новой функции

Любая функция оформляется планом в папке `plans/`.

- Один план = одна функция. Если план уже есть — работаем с ним
- Имя файла: `YYYY-MM-DD-название-функции.md`
- План делится на фазы. У каждой фазы статус `[ ]` или `[x]`
- В конце плана — итоговый блок: реализован целиком или нет, что осталось
- Любой агент обязан актуализировать план после каждой сессии

## ВАЖНО: исправление багов — корневая причина, а не симптом

- Не лечи симптом, не найдя корневую причину
- Чини на уровне источника правды (owner layer), а не там, где симптом проявился
- Избегай компенсации в дочернем слое (fallbacks, патчи, дублированная логика, ветвления)
- Перед фиксом — ультра-глубокое сквозное исследование системы:
  - сверху-вниз: route → page → container → orchestration → state
  - снизу-вверх: function → hook → service → API → DB
- Диагностика по слоям: data/contracts → business logic → async/timing → UI state → integration → architecture
- Если баг проявляется в дочернем слое — сначала проверь родительский (owner) слой
- При изменении механики выровняй все напрямую связанные слои:
  contracts, handlers, queries, cache, serializers, loading/error states
- Будь скептичен к фиксам в один файл — обоснуй, почему остальные слои не задеты
- Для frontend-багов исследуй полный поток: route → layout → page → hooks → API → backend
- Предпочитай системные фиксы, но соразмерные масштабу проблемы
- Если нужна ре-архитектура — определи scope, риски, совместимость и порядок раскатки

### После каждого фикса бага — записать урок

После любого исправления бага записывай отдельный файл-урок в `docs/lessons/YYYY-MM-DD-краткое-название.md`.

Формат урока:
- **Симптом** — как баг проявлялся
- **Корневая причина** — что на самом деле сломалось и в каком слое
- **Почему так получилось** — как пропустили, какая абстракция/контракт оказались дырявыми
- **Как починили** — на каком слое, какие связанные слои выровняли
- **Что проверить в будущем** — паттерн или сигнал, по которому ловить похожие баги

## ВАЖНО: завершение каждого чата

В конце каждой сессии записывай рефлексию в файл `.business/история/YYYY-MM-DD-краткое-название.md` (создавай папку при необходимости).

Формат:
- Какая задача была поставлена
- Как я её решал
- Решил ли — да / нет / частично
- Эффективно ли решение, что можно было лучше
- Как было и как стало
