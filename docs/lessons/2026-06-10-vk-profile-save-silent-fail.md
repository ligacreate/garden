# Урок: профиль не сохранялся (VK) + молчаливый ложный успех

**Дата:** 2026-06-10
**Файлы:** `lib/contactNormalize.js`, `services/dataService.js`, `views/UserApp.jsx`

## Симптом
Пользователь сохраняет профиль с заполненным VK → данные не сохраняются, но UI
показывает успех. Молчаливый ложный успех, никакого сигнала об ошибке.

## Корневая причина — два независимых бага

1. **Нормализация контактов не срезала суб-хост.** `normalizeVk` срезала протокол
   `https?://`, но префиксы `www.` и мобильный `m.` оставались и попадали внутрь
   итоговой ссылки: `https://www.vk.com/vasya` → `https://vk.me/www.vk.com/vasya`
   → не проходит `VK_RE` → «невалидный VK». Браузер/телефон часто отдают ссылку
   именно с `www.`/`m.`, поэтому по сути валидный VK падал в invalid.
   Тот же баг был в `normalizeTelegram` (`https://www.t.me/x`).

2. **`updateUser` глушил ошибку.** Блок PATCH профиля ловил исключение в `catch`,
   писал `console.warn` и **проваливался дальше** к `return updatedUser` — функция
   возвращала успех, хотя PATCH не прошёл (или валидация бросила `throw`).
   Вызывающий `UserApp.handleUpdateProfile` имеет корректный `catch` + `onNotify`,
   но ошибка до него не доходила.

## Почему так получилось
- Нормализатор обрабатывал только один вид «мусора» (протокол), молчаливо
  предполагая, что хост всегда голый `vk.com`/`t.me`. Дырявый контракт: «вход уже
  без суб-хоста».
- `catch` без `throw`/`return` — классический «проглоченный» обработчик. Соседний
  блок role/status в той же функции **пробрасывал** ошибку, профильный — нет.
  Несогласованность внутри одной функции = сигнал.

## Как починили
- **Owner-слой нормализации** (`contactNormalize.js`): срезаем `^(www|m)\.` сразу
  после протокола, до обработки хоста — в обеих функциях (VK и TG, параллельный
  баг того же типа). Логины VK/TG не содержат точку, поэтому легитимный username
  не пострадает.
- **Owner-слой сохранения** (`dataService.updateUser`): `catch` теперь
  `throw e` — не выдаём ложный успех. Валидационные ошибки помечены флагом
  `err.userFacing = true`.
- **UI-слой** (`UserApp.handleUpdateProfile`): `onNotify(e?.userFacing ? e.message
  : "Ошибка сохранения профиля")` — пользователю конкретный текст валидации,
  а технические ошибки PATCH — общий (не вываливаем сырьё).

## Что проверить в будущем
- **Нормализаторы свободного ввода**: прогонять реальные варианты, которые отдаёт
  платформа (с `www.`, `m.`, протоколом, `@`, trailing slash). Не предполагать
  «голый» хост.
- **`catch` без `throw`/`return`** в функции, которая возвращает «успех» ниже —
  красный флаг молчаливого ложного успеха. Особенно если соседний блок той же
  функции ошибку пробрасывает.
- Оптимистичный `onUpdateUser(updated)` срабатывает ДО `await` — при ошибке
  локальный стейт временно расходится с БД. Отдельный потенциальный backlog-пункт
  (откат оптимистичного апдейта при сбое), вне scope этого фикса.
